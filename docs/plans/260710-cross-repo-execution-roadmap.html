エグゼクティブサマリ

4つのプラン（AnalogBoard 再構築・下流改変・gcsa ストア対応・sys_app 直読＋デスクトップ化）を最短で完走させる実行順を定める。 鍵は「最初に動くのは AnalogBoard ではない」こと——AnalogBoard フェーズ0 の暗号化込み往復プロトタイプは G4 コーデック登録済みの gcsa が改変なしでロードできることが合格条件のため、gcsa フェーズA が最初のクリティカルパスになる。 実機確認（2026-07-13 週予定）までの今週が、gcsa フェーズA と sys_app バグ修正に充てる最適な時間である。

絶対に守る順序は2つだけ：① gcsa フェーズA → AnalogBoard フェーズ0 プロトタイプ、 ② gcsa の ML 直列化中立化 → sys_app ML/DR 実装（sklearnex 形式のモデル資産を作らせない）。 残りは2つの同期点（出力契約凍結・フェーズ3.5 往復ゲート）で自然に同期する。 sys_app フェーズ2（直読）を AnalogBoard フェーズ1〜3 と並行で済ませておくことで、フェーズ3.5 は「新規実装」ではなく「検証だけ」になる——これが全体最短の要。

直列実行順（一人で進める場合）

並行できない場合は次の順で直列化する。太字が同期点。

  1. sys_app バグ3件修正（小さく独立・すぐ終わる）
  2. gcsa フェーズA（G4・fl_waveform・マーカ first-class・スキーマ API）——実機確認の週までに
  3. AnalogBoard フェーズ0（実機確認＋プロトタイプ＋出力契約凍結）
  4. gcsa フェーズB（ML 中立直列化を先頭に → G2・バリデータ・並行性）
  5. sys_app ML/DR 実装
  6. AnalogBoard フェーズ1（ネイティブコア）——最長。区切りの良い所で 7 を挟む
  7. sys_app フェーズ2（完全直読）＋CI コーパス
  8. AnalogBoard フェーズ2〜3（UI/描画・HIL）
  9. AnalogBoard フェーズ3.5（直読往復ゲート）→ UX-16 レベル判断
  10. sys_app Tauri デスクトップ化
  11. AnalogBoard フェーズ4（実機受け入れ・Win11）

同期点とゲート

同期点いつ確定するもの合格条件
合流点1：出力契約凍結 AnalogBoard フェーズ0 末（実機確認の週） D21 字段名・シャーディング方式・暗号ワイヤ形式・ストアスキーマ（fl_waveform 込み） 暗号化込み往復プロトタイプが G4 登録済み gcsa で改変なしロード＋D21 字段が gcsa meta round-trip で保存される
合流点2：フェーズ3.5 直読往復ゲート AnalogBoard フェーズ3 後 デュアル・シームレスの成立（D11） ストア（暗号化・finalized）→ gcsa 無再計算 → sys_app registration-only＋直読 → scatter/gate 統計 bit 一致・再計算スパイ0回・複製ゼロ。S10/G5 で CI 常設化
絶対に守る順序は2つだけ
  • ① gcsa フェーズA → AnalogBoard フェーズ0 プロトタイプ：G4/fl_waveform/マーカが無いと往復検証が成立しない
  • ② gcsa ML 中立直列化 → sys_app ML/DR：sklearnex 形式のモデル資産を作らせない（作ってしまうとデスクトップ化時に oneDAL 同梱 or 全再学習の二択になる）

この2つを守れば、残りは合流点1・2で自然に同期する。

リスクと注意

Highgcsa フェーズA が実機確認週に間に合わない
往復プロトタイプが「gcsa 側の未実装」で止まり、契約凍結が遅れて全下流が待ちになる。対策：今週の最優先を gcsa A に置く。間に合わない場合はプロトタイプを平文で先行実施し（レイアウト適合のみ検証）、暗号往復は追試とする——ただし合格判定（凍結）は暗号込みが揃うまで下さない。
Mediumシャーディング決定の手戻り
round-robin の数式はバリデータだけでなく index-gather 経路・interleave 逆写像にも焼き込まれており（2026-07-10 検証）、凍結後の変更はリポジトリ横断の大改修になる。対策：フェーズ0 の往復検証で必ず決着させ、gcsa バリデータ改修（G5）と同一の契約変更として扱う。
Mediumsys_app 直読の後回し
ステージ④を先送りして AnalogBoard フェーズ3.5 に到達すると、3.5 が「検証」ではなく「実装待ち」になりゲートが長期停滞する。対策：凍結後は sys_app フェーズ2 を AnalogBoard フェーズ1〜3 と並行で必ず進める（プロトタイプ産ストアで開発可能）。
Lowリソース競合（一人開発時）
3トラック並行はチーム前提。一人の場合は「直列実行順」に従い、sys_app Tauri 化（最も独立性が高い）を緩衝材として後ろへずらす。
Assumptions
  • 実機確認は 2026-07-13 週に実施され、フェーズ0 ゲート（実機セッション完了・コーパス取得・往復成立・契約凍結）を満たす。
  • 工数・体制は未確定のため本書は依存順序のみを定め、日付は同期点の目安に留める（親プランのフェーズゲート基準が正）。
