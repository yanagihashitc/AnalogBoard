概要・実行前提

暫定・書き換え前提（必読）

本ドキュメントは暫定であり、確定版ではない。以下を前提とする：

  • 実行タイミング：この改変は AnalogBoard 側の改変（新アプリの出力実装）が終わった時点で実行する。それまでは着手しない。
  • 書き換えが起きる：AnalogBoard 側の実装確定に伴い、出力契約（Zarr スキーマ、meta.json フィールド、特徴量の順序/名前、精度、gate 形式）はまだ変わりうる。したがって本プランの各項目（特に file:line・フィールド名・型）は実行時に必ず再検証する。
  • スナップショット：本プランの sys_app/gcsa への参照（file:line・関数名）は現時点のソースに基づくスナップショット。実行前に該当箇所を再確認すること。
  • 直接改変しない：プロジェクトルールにより gcsa/sys_app は本作業では直接編集せず、本書は改変プロンプトとして提示する。実行は各リポジトリの担当が行う。

AnalogBoard 再構築（親プラン D5/D9/D10/D11/D19/D21/D22）で、本アプリは A/H/W を算出してgcsaネイティブ Zarr ストア（測定配列チャンクは D19 で暗号化）に記録し、gcsa と sys_app の両方でシームレスに使えることを要件とする。gcsa は構造上ほぼシームレス（fl_waveform とマーカ意味論の追加が必要）。sys_app には(1) gcsa ストアのリーダが無い、(2) 取り込み時に A/H/W を無条件再計算する、という2ブロッカーがある。 解は ストア完全直読（D22・2026-07-10 転換）：import はストアの meta.json＋配列形状から Dataset 行を登録するだけ（registration-only・データ複製なし）とし、特徴量・波形の読取は gcsa 経由でストアを直接読む。（旧解「射影コピー」は、①ピーク RAM≒生データサイズで大データセット import 不可、②読者不在の全波形コピー、③D19 暗号化を平文コピーが無効化、の3問題により廃止——経緯は AnalogBoard/docs/260709-analogboard-sysapp-integration-study.html）

実行順序と依存関係

本改変は AnalogBoard 側の出力契約が凍結されてから着手する。sys_app（S）と gcsa（G）は契約凍結後に並行可能で、最後にデュアル・シームレス往復テストで締める。

依存（2026-07-10 改訂：凍結の3段階化と着手可能範囲の明確化）

契約確定は3段階で扱う：① Contract RC（本プランの固定入力契約表＝設計意図。現状）→ ② joint validation（フェーズ0：C++ライタ×gcsa リーダの暗号化込み往復プロトタイプで RC を実証）→ ③ Frozen v1（合流点1。以降の変更は版上げ＋変更管理）。着手規則：G1/G3/G4（契約凍結前必須と記した項目）は RC に対して先行着手してよい——これらは②の前提であり、②で判明した差分は③前に吸収する。それ以外の S/G は③（Frozen v1）後に着手する。旧文言「凍結されるまで S/G は着手しない」は G1/G3/G4 の先行必須と矛盾していたため上記に置換。スキーマ変更が③後に発生した場合の再凍結フロー（版上げ→コーパス更新→両リーダ再検証）は変更管理節と gcsa プラン C-9 に従う。

Part A — sys_app 改変（直読版：S1,S3,S4,S6〜S11。S2/S5 は廃止）

各項目は改変プロンプト形式（what / why / how / verify）。file:line は現ソースのスナップショット、実行時に再検証する。2026-07-10 改訂（D22）：S1 を「射影コピー」から「registration-only import＋直読リーダ」へ再設計し、射影 ETL 前提だった S2/S5 を廃止（不変条件は直読リーダに内在化）、S9〜S11 を追加した。実装詳細は sys_app リポジトリの実装プラン（sys_app/docs/plans/260710-analogboard-store-and-desktop-plan.html）に展開する。

S1. registration-only ストア取り込み＋直読リーダ（旧：射影 → 2026-07-10 再設計）

What(a) registration-only import：ストアの meta.json＋配列形状から Dataset 行（event_count・channels_json・store パス）を登録するだけにする——データ複製なし・status=finalized のみ対象。(b) 直読リーダ：特徴量読取を DuckDBFeatureStore（Parquet）から gcsa 経由の ZarrFeatureStore（復号・パーティション連結・event_idx 合成、返り値 dict[str, np.ndarray] 互換）へ差替える。bin 取り込みは互換で残す。
WhyD22。旧射影は①全計測 RAM ロード（ピーク RAM≒生データサイズ、1M イベント≒62GB で import 不可）②読者不在の全波形コピー（data.zarr、dataset_import_steps.py:787-822 に書くが読むコード無し）③D19 暗号化を平文コピーが無効化、の3問題。sys_app の特徴量読取は DuckDBFeatureStore 1箇所（duckdb_feature_store.py:100-148・呼出毎 in-memory 接続・read_parquet(path)）にチョークしており、コンシューマは gate（gate_mask_operations_mixin.py:112）と可視化（visualization_pulse_event_source.py:375）の2モジュールのみ——同型 dict を返せば上位は無改修。
Howimport：import_from_path で store 判定（.gcsa_store.json/meta.json 有）→登録専用ステップへ分岐（既存 build_importing_dataset/run_pipeline_in_transaction/メタ読取パターン dataset_import_persistence_service.py:271-284 を再利用）。リクエストスキーマはストアパス指定・スキャン/発見はストアベース・idempotency はストア指紋（meta.json の dataset id＋write_generation。現行の全ファイル内容 SHA-256 方式（dataset_import_input_service.py:185-189）は数十GB級ストアで非現実的なため置換）。channels_json のレンジは meta の min/max（D21）から取得（全行スキャン不要）。display_name/comment/tags は表示名に採用。読取：DuckDBFeatureStore と同一シグネチャの ZarrFeatureStore を実装し2箇所の生成部で差替え（gcsa の get_pulse_features(features=[...])・列指定読取を使用）。extract_pulse_features は一切呼ばない。 登録ライフサイクル（2026-07-10 追加）：(a) 同一 dataset_id の再登録は冪等 upsert（write_generation が進んでいれば更新・派生キャッシュ無効化。DB に source_kind／source_dataset_id／store_path／store_format／feature_schema_version／write_generation 列＋一意制約を追加——Idempotency-Key 任意設定だけでは二重登録を防げない）。(b) パス検証は既存の DATA_DIR 制約（validate_path_within_data_dir）を継承し allowed roots を設定化、symlink/UNC/コンテナパスの正規化規則を明記（親プラン「データ移送契約」の対応表と対）。(c) ストアの移動・削除・オフライン時＝該当データセットを unavailable 表示にし fail-loud（自動削除しない）、再登録（パス更新）で復帰。(d) meta.json のマニフェストと実配列 shape の不一致＝登録拒否（明示エラー。切詰め読みで隠さない）。(e) 登録はソースへ一切書かない——現行 import ルートは bootstrap_sys_app_directories() で source 直下に sys_app/{store,artifacts,gates}/ を作成する（routers/datasets.py:407-409・export 経路 export_tasks.py:419-423 にも同型）ため、store 経路では呼出し禁止。派生物は app 管理の derived root へ（S9 参照）。(f) 既存 endpoint は cfg_path 必須（datasets_import_helpers.py:63-78）のため、store 登録は tagged union か別 endpoint とする。
Verifystore のみのディレクトリの import が秒未満で完了し Dataset 行が正しい（event_count==N・channels_json レンジ==meta の min/max）。PulseFeatureExtractor.extract_all 不呼出（spy）。scatter/gate クエリ値がストアの pulse_features と bit 一致。status=open のストアは import 拒否（明示エラー）。バルクコピー（parquet/data.zarr/waveforms.json）が生成されないこと。

S2. 廃止（2026-07-10・直読転換に伴い解消）

廃止の経緯

旧 S2「特徴量算出を記録済みが無い時のみに条件化」は、import が何もデコードしなくなる registration-only 化（S1）で対象コードパス自体が消えるため廃止。不変条件（D5：記録済み A/H/W を黙って上書きしない）は直読リーダ（S1b）に内在化され、受け入れテストの再計算スパイ（0回）で検証する。レガシー bin import 経路の無条件再計算（dataset_import_steps.py:239）は互換経路として残るが、ストア経路とは独立。gcsa 側の同じ不変条件は G2 が担う。

fail-loud 要件（2026-07-10 追加・実コード検証で確定）：現行 sys_app には読取失敗→gcsa 再抽出→Parquet 永続化の fail-open フォールバックが実在する（visualization_event_source.py:144-151 → visualization_pulse_event_source.py:287-361 の recover_events_from_gcsa が extract_pulse_features を呼び :355-360 で永続化。読取例外 FileNotFoundError/ValueError は :377-379 で黙って None 化。波形にも同型のフォールバックあり：visualization_waveform_source.py:170-190）。ストア由来データセット（storage=gcsa_store）ではこの経路を構造的に無効化し、読取失敗は明示エラー（fail-loud）とする——復号失敗・パーティション不整合を黙って再計算で覆い隠すのは D5/D19 の同時違反。受け入れに「extract_pulse_features／recovery 永続化／波形再生成 fallback の呼出し 0 回」の spy を含める（S10 の CI でも常時検証）。

S3. channels_json の名前・順序・レンジを meta.json から決める（直読版に改訂）

Whatチャンネル名と順序を meta.json の宣言リストから、値レンジを meta.json の min/max（D21）から決める（dict 挿入順・全行スキャンに依存しない）。
Why位置行列(N,24)に名前が無く、現状は dict キー順から推定（dataset_import_steps.py:286-302）。誤ラベルは gate 軸検証（GateAxisMismatchError）と統計参照（routers/gates.py:298-315）を壊す。レンジは現状 import 時の全行スキャンで計算しており、registration-only 化とは両立しない——D21 の min/max メタが代替。
Howmeta の順序名を読み、列数と一致をアサートし、build_pulse_feature_channels（persistence 81-98,170-207）へ名前＋min/max を渡す。ストア宣言名 == channels_json となる検証を追加。
Verifychannels_json が meta の順序・レンジと一致。FSC_A/SSC_H の gate が解決し統計が出る。登録時に pulse_features 本体を読まない（I/O 監視）。

S4. GMI 波形の直読化（旧：waveforms.json 生成 → 2026-07-10 再設計）

What波形表示・export をストアの gmi_waveform(N,5,2400) の直接スライス（gcsa get_gmi_waveform(event_indices=...)）に切替え、import 時の waveforms.json 生成を廃止。プレビューと export を分離（2026-07-10 追加）：UI プレビューのみ 128 イベント上限（GMI_WAVEFORM_MAX_COUNT=128）を維持し、export／API は全イベント範囲をチャンク順に直読する（ストリーミング/ページング）。注：現行実装は export も同じ 128 上限チェーンを通っており（export_tasks.py:166-171 → resolve_waveform_source の3ソース全てが 128 cap——既存の実質バグ）、直読化はこの混同を解消する機会であって固定化してはならない。
Why現行 waveforms.json は 128 イベント上限の JSON プレビューで、生成には全計測ロードが要る（registration-only と非両立）。読取チェーンは1本（resolve_waveform_source、visualization_waveform_source.py:193-220）・コンシューマ2箇所（波形エンドポイント・export）で差替えは小さい。誤順は GMI プロットを黙って取り違える。
Howresolve_waveform_source にストア直読分岐を追加し、取得した配列を既存 build_multi_channel_waveform_payload（waveform_payload.py:141-215・array-like を受ける）へ渡す。resolve_waveform_store_dir の dataset.zarr_path 依存（visualization_waveform_helpers.py:26-34）をストアパス項目に置換。meta の GMI 順 == FH_CHANNEL_NAMES をアサート。
Verify波形エンドポイントが 5 ch を返し、既知イベントの各サンプルがストアの gmi_waveform と一致（復号込み）。import 後に waveforms.json が存在しない。

S5. 廃止（2026-07-10・registration-only 化に伴い解消）

廃止の経緯

旧 S5「ストアモードでの入力検証の緩和」（cfg/bin 任意化・byte 検査スキップ・event_count をストアから導出）は、ストア経路が bin パイプラインを通らなくなる S1（registration-only）に吸収され独立項目として消滅。event_count はストア meta＋マニフェスト（D21）から取得し、pulse_features.shape[0] 依存による「取得中 import のサイレント切詰め」ハザードも status=finalized ガードで同時に解消される。レガシー bin import の検証は無変更で残す。

S6. 精度ポリシーの確定（直読版で適用範囲を更新）

備考2026-07-10：ストアデータセットの特徴量読取は直読（S1b・float64 のまま）になるため、本項の主対象は残る内部面——読取パスの round(6) 除去（直読にも効く）・レガシー bin import の Parquet・補償 sidecar（S9）の dtype——に更新。往復 bit 一致の判定基準は不変。
What確定（親プラン D15）：Parquet 列を float64 化し、読取の round(6) を除去する。

G4. チャンク保護（AEAD Store ラッパ）【新規 2026-07-10・D19・契約凍結前必須。2026-07-10 方式変更】

What方式変更（2026-07-10・実証済み）：numcodecs の Codec は encode(buf)/decode(buf) のみでチャンク座標を受け取らない（numcodecs 0.13.1 abc.py:41-59／zarr 2.18.3 core.py の _encode_chunk/_decode_chunk）。したがって当初案「カスタム numcodecs コーデック gcsa-aes」では座標を AAD に束縛できず、チャンクファイルの入替えを検知できない。読取側の AEAD を zarr の Store ラッパ層（BaseStore.__getitem__／__setitem__ がチャンクキーを受け取る）に移す。.zarray の compressor は内側の Bloscを宣言する。
WhyD19。AnalogBoard の C++ ライタが書く暗号化チャンクを gcsa（＝sys_app の直読層）が復号できる必要がある。抑止レベル（バイナリ解析には耐えない）を明記した設計。GCM 認証タグはチャンク単位の破損検知を兼ねる。
Howgcsa の配列 open（_zarr_atomic.py の open_zarr_v2_array／open_existing_zarr_v2_array）が生パスの代わりにAEAD Store ラッパを構築するよう変更する。ラッパはルートパスから dataset_id／配列相対パスを、キーからチャンク座標を得て AAD を組み立て、.zarray/.zattrs 等の非チャンクキーは素通しする。C++ ライタは従来どおりチャンクキー＝ファイル名で直接書くため、両者は同一の AAD を計算でき相互運用する（DirectoryStore のキー↔ファイル名対応で実証）。鍵は key_id 引きの key provider に隔離（将来の外部鍵化の差替え点）。ワイヤ形式・AAD のバイト表現・nonce 一意性方式を C++ 側と機械可読仕様で固定し、C++／Python の KAT を Must。zarr の blosc partial-decompress は禁止（全チャンク復号前提）。numcodecs.register_codec による独自 ID 方式は採らない。
VerifyC++ ライタ産の保護済みストアが、AEAD ラッパ経由の gcsa で無改変ロード（フェーズ0往復）。実 zarr 上で2座標のチャンクファイルを丸ごと入替えると必ず失敗するテストを RC ゲートに含める（AAD 座標束縛の実証）。tag 改竄・wrong key・未知 key_id・切詰めチャンクはすべて fail-loud。ラッパ非経由の素の zarr.open_array では Blosc 復号が失敗し、平文が得られないこと。AAD のバイト表現（区切り・座標整数表現・エンディアン）を C++／Python の KAT で byte-exact に固定。

再計算スパイ 0 回、登録がデータ複製ゼロ・秒未満、scatter/gate 値がストアと bit 一致、status=open の import が明示拒否、gate 往復が無損失、ストア本体が下流から一切書き換えられない。これが満たされれば「gcsa と sys_app でシームレス」を担保する。本テストは S10/G5 により CI コーパスとして常設化する。

受け入れマトリクス（2026-07-10 追加・「各1件」からの拡張）

上記の基本往復に加え、コーパスと受け入れは次の軸を掛け合わせてカバーする（S10/G5-c の CI マトリクス）：

  • ストア形態：fl_waveform ON／OFF・単一パーティション／複数パーティション（再アーム追記済み）・部分チャンク（端数行）
  • 状態：status=open（拒否）／finalized・マニフェスト欠損／shape 不一致（登録拒否）
  • 暗号 negative：wrong key・タグ改竄・チャンク swap・未知 key_id・コーデック未登録環境（すべて fail-loud）
  • fail-open 防止：extract／recovery 永続化／波形 fallback の呼出し 0 回（spy）
  • gate 意味論：全 gate 型×境界上イベント＋全象限のフィクスチャで gcsa／sys_app の mask・統計が一致
  • 補償 sidecar：適用→表示→除去→再適用のライフサイクル・write_generation 変化時の無効化・ストア本体不変（mtime/ハッシュ）
  • スケール：大規模データセット（目安 1M イベント）で登録が秒未満・scatter 初回描画が実用時間内・export 全件が完走
  • レガシー互換：既存 bin import データセットの表示・gate・export に退行がない
