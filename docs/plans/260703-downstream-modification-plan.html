概要・実行前提

暫定・書き換え前提（必読）

本ドキュメントは暫定であり、確定版ではない。以下を前提とする：

  • 実行タイミング：この改変は AnalogBoard 側の改変（新アプリの出力実装）が終わった時点で実行する。それまでは着手しない。
  • 書き換えが起きる：AnalogBoard 側の実装確定に伴い、出力契約（Zarr スキーマ、meta.json フィールド、特徴量の順序/名前、精度、gate 形式）はまだ変わりうる。したがって本プランの各項目（特に file:line・フィールド名・型）は実行時に必ず再検証する。
  • スナップショット：本プランの sys_app/gcsa への参照（file:line・関数名）は現時点のソースに基づくスナップショット。実行前に該当箇所を再確認すること。
  • 直接改変しない：プロジェクトルールにより gcsa/sys_app は本作業では直接編集せず、本書は改変プロンプトとして提示する。実行は各リポジトリの担当が行う。

AnalogBoard 再構築（親プラン D5/D9/D10/D11）で、本アプリは A/H/W を算出してgcsaネイティブ Zarr ストアに記録し、gcsa と sys_app の両方でシームレスに使えることを要件とする。gcsa は構造上シームレスだが、sys_app には(1) gcsa ストアのリーダが無い、(2) 取り込み時に A/H/W を無条件再計算する、という2ブロッカーがある。解は射影（projection）：sys_app は自前の Parquet/JSON を内部派生面として保ちつつ、取り込み時に gcsa ストアから射影して生成し、再計算しない。これにより可視化/gating/compensation は無改修で D5 を満たす。

実行順序と依存関係

本改変は AnalogBoard 側の出力契約が凍結されてから着手する。sys_app（S）と gcsa（G）は契約凍結後に並行可能で、最後にデュアル・シームレス往復テストで締める。

依存

「固定入力契約」（次節）が凍結されるまで S/G は着手しない。凍結前に着手すると、契約変更のたびに手戻りが発生する。

固定入力契約（AnalogBoard 側が凍結する）

下流改変の前提となる、AnalogBoard 出力の固定契約。これらは現時点の設計意図であり、実装確定で変わりうる（変更管理の節参照）。実行時点の確定値に置換すること。

Part A — sys_app 改変（S1〜S8）

各項目は改変プロンプト形式（what / why / how / verify）。file:line は現ソースのスナップショット、実行時に再検証する。

S1. gcsa ストア取り込みモード（射影）

Whatgcsa ストアを読み、pulse_features/gmi_waveform/(fl_waveform)/meta.json を sys_app 既存面（sys_app/store/pulse_features_{id}.parquet・GMI waveforms.json・channels_json）へ射影。bin 取り込みは互換で残す。
WhyD9 の主出力はストアだが sys_app にリーダが無く（ZarrStore/gcsa_store 参照ゼロ）、権威面は Parquet（dataset_import_steps.py:470-493、duckdb_feature_store.py:41）。射影で可視化/gating/compensation を温存し D5 を満たす。
Howimport_from_path（routers/datasets.py:294-325）で store 判定（.gcsa_store.json/meta.json 有・_cfg.txt 無）→新ステップへ分岐。ストアを開き pulse_features(N,24) と meta の順序名を読み、_persist_pulse_features_parquet（dataset_import_steps.py:470-493）へ event_idx＋名前付き float 列で渡す。extract_pulse_features を呼ばない。meta に display_name/comment/tags（任意・additive、親プラン UX-17）があれば Dataset の表示名・説明に採用する（無ければ従来命名）。
Verifystore のみのディレクトリを import → 24 正名列・N 行の Parquet 生成、PulseFeatureExtractor.extract_all が呼ばれない（spy）、scatter/gate クエリ値がストアと一致。

S2. 特徴量算出を「記録済みが無い時のみ」に条件化

Whatimport と recovery の両方で、記録済み特徴量があれば再計算しない。
WhyD5：記録済み A/H/W が権威。現状 dataset_import_steps.py:239 と visualization_pulse_event_source.py:314 が無条件に再計算し記録値を上書き。
How_load_measurement_and_pulse_features（dataset_import_steps.py:215-249）と recover_events_from_gcsa（visualization_pulse_event_source.py:287-315）に、ストア/Parquet の特徴量があればそれを返し、無い時だけ extract_pulse_features を呼ぶ分岐を追加。
Verify抽出器の出力と意図的にずらした値をストアに書き、sys_app の scatter/gate がその「ずらした（記録）」値を返す（再計算でない）ことを確認。

S3. channels_json の名前・順序を meta.json から決める

Whatチャンネル名と順序を meta.json の宣言リストから決める（dict 挿入順に依存しない）。
Why位置行列(N,24)に名前が無く、現状は dict キー順から推定（dataset_import_steps.py:286-302；persistence 137 で event_idx 除去）。誤ラベルは gate 軸検証（GateAxisMismatchError）と統計参照（routers/gates.py:298-315）を壊す。
Howmeta の順序名を読み、列数と一致をアサートし、Parquet ライタと build_pulse_feature_channels（persistence 81-98,170-207）に渡す。ストア宣言名 == channels_json となる検証を追加。
Verifychannels_json が meta の順序と一致。FSC_A/SSC_H の gate が解決し統計が出る。

S4. GMI 波形 JSON を単一 gmi_waveform から生成（順序ガード）

Whatストアの単一 gmi_waveform(N,5,2400) から GMI 波形 JSON を生成し、チャンネル順をガード。
WhyD9 は単一 gmi_waveform。現状は cfg 経由で fh_waveforms を再読込（dataset_import_steps.py:531-602,917-951）。誤順は GMI プロットを黙って取り違える（waveform_payload.py:141-193）。
Howストアの gmi_waveform を build_multi_channel_waveform_payload（waveform_payload.py:141-215）へ。meta の GMI 順 == FH_CHANNEL_NAMES（fsGMI,ssGMI,flGMI,dGMI,bfGMI、waveform_payload.py:7-16）をアサート。waveforms.json/waveform_meta.json 形状は維持。
Verify波形エンドポイントが 5 ch を返し、既知イベントの各サンプルがストアの gmi_waveform と一致。

S5. ストアモードでの入力検証の緩和

Whatストアがある場合 _cfg.txt＋fh/fl.bin を任意化し byte サイズ検査をスキップ、event_count はストア形状/meta から。
WhyD9 で bin は任意互換だが、現状 import は cfg 必須（dataset_import_input_service.py:84-90）・FH/FL byte 検査（dataset_import_steps.py:605-695；routers/datasets.py:347-365、FH_BYTES_PER_CELL=5*2400*2 / FL_BYTES_PER_CELL=8*2400*2）・event_count を bin から導出。
Howanalyze_binary_step と byte 検査をストア有無で分岐、event_count = pulse_features.shape[0]（persistence 150-168 の pulse_feature_event_count）。
Verifystore のみ（cfg/bin 無）の import が完走し、Dataset 行の event_count == N。

S6. Parquet 精度ポリシーの確定

What確定（親プラン D15）：Parquet 列を float64 化し、読取の round(6) を除去する。
WhyD9 は float64、sys_app Parquet は float32（dataset_import_steps.py:488）＋読取 round(6)（visualization_service_helpers.py:47）で gcsa と bit 一致しない。
HowParquet 列を float64 化し round(value,6) を除去（float32＋許容差案は不採用）。
Verify抜取値がストアと選定精度で一致。読取が float coerce でエラー無し（duckdb_feature_store.py:67）。

S7. gate 種の整合＋ gcsa-clean v1 エクスポート派生

Whatgcsa all を sys_app 仮想 root にマップ、sys_app histogram-range の下map、gcsa-clean v1 エクスポート派生を追加。
Whygcsa v1 の all は import で GateInvalidTypeError（import_mapping.py:100-106）；sys_app histogram-range は gcsa v1 に無い；既定 export は v2.0＋fill_enabled/fill_order＋トップ階層（export_service.py:64-100）で strict gcsa v1 と不一致。
Howimport で all → FALLBACK_GATE_ID（gate-root）にマップ（例外にしない）。{version:'1.0', gates:[gate_id,display_name,color,type,axes,transform_space,params,parent_gate_id]} のみを出す export 派生を追加（fill_*/階層を除去）。
Verifyall を含む gcsa v1 ファイルが例外なく import。gcsa-clean export が gcsa へ無改変で再 import。

S8. 空間トークンを meta.json から取得＋色正規化

What特徴量空間トークンを meta.json から取り、gate 色を #RRGGBB に正規化。
Whysys_app は軸/binding の空間を pulse_features 既定（schemas/gate.py:162；routers/gates.py:305）。ストアが別トークンなら bindings.space がずれる。非HEX色は作成で拒否（HEX_COLOR_PATTERN gate.py:38）または import で再パレット化。
Howmeta の空間トークンを軸/binding 既定に使用。作成/import 前に色正規化（alpha 除去/blend、#RGB 展開）。
Verifybindings.space がストア宣言トークンと一致。色が gcsa→sys_app→gcsa 往復で #RRGGBB を保つ。

Part B — gcsa 改変（G1〜G3）

G1. 正準ストアスキーマの凍結・公開

WhatAnalogBoard が書き sys_app が射影する正準ストアスキーマを凍結し公開する。
Why両側が同じ契約を参照する単一の真ソースが必要（順序名・GMI名順・dtype/shape）。
How版付きスキーマ文書＋、正準の順序特徴量名（== PulseFeatureExtractor.extract_all の 8ch×{_A,_H,_W}）・GMI 名順（== FH_CHANNEL_NAMES）・配列 dtype/shape を返す小 API を追加。
VerifyAPI 出力が実 extract_all の並びと一致。文書の版が meta.json versions と対応。

G2. ストア読取で記録済み特徴量を権威にする（無再計算）

Whatストア読取・解析が記録済み pulse_features を権威として返し、既存時に黙って再計算しない。
WhyD5：アプリ算出が権威。gcsa 側の再計算が既定だと記録値と食い違う恐れ。
Howストアリーダは格納 pulse_features をそのまま返し、再計算は明示 opt-in にする。
Verifyずらした値を含むストアを開き、解析が記録値を返す（再計算しない）。

G3. fl_waveform リーダ追加＋ gate モデル整合

What任意 fl_waveform(uint16 (N,8,2400)) を ZarrStore 読取に追加（gmi_waveform 対称）。加えて gate モデル整合（histogram-range、all/root、余剰キー許容、色正規化）。
WhyD10：生FL保全でアルゴリズム改訂時に再計算可能に。現状 ZarrStore は算出特徴量＋生FH のみ保持。gate 型集合が sys_app と非一致。
How_gmi_waveform_path/get_gmi_waveform/append_gmi_waveform を _fl_waveform_path/get_fl_waveform/append_fl_waveform として鏡像化（無ければ従来挙動）。chunk (2000,None,None)。gate は histogram-range 受理 or range/threshold への合意下map、all/root の対応、色正規化を合意。
Verifyfl_waveform 有りのストアが既存フローで無改変ロード、get_fl_waveform が (N,8,2400) uint16・FL 順 FSC,SSC,FL1..FL6 で往復。

Part C — gate すり合わせ seam

gate は「取得データ」とは別の合意事項。gcsa v1 と sys_app で gate 型集合が異なるため、往復（AnalogBoard→gcsa→sys_app→戻し）が壊れない対応表を合意する。

種別gcsa v1sys_app合意方針（案）
polygon/rectangle/threshold/quadrant/booleanありありそのまま往復可
all（root）あり無（仮想 root gate-root）import 時 all → 仮想 root にマップ、正準 export では root を出さない
histogram-range無ありgcsa に受理 or range/threshold へ下map を合意（S7/G3）
ellipse不可（拒否）不可UI の範囲選択に ellipse を出さない
version/余剰キーv1.0・厳格v2.0＋fill_*＋階層gcsa-clean v1 export 派生（余剰キー除去）＋余剰キー許容の合意
色#RRGGBB#RGB/#RRGGBB#RRGGBB に正規化
前提の確認

確定（親プラン D13）：AnalogBoard は gate を正準v1のみで出力する。したがって本対応表を確定し、gate 往復テストを受け入れに組み込む。なお親プラン UX-03（gate作図時の互換フィードバック）により、AnalogBoard の作図UIは正準v1で表現できる gate 種のみ提示する方針（本表「ellipse を出さない」の一般化）。sys_app 側で生成される histogram-range の下map（S7/G3）は引き続き必要。

受け入れ（デュアル・シームレス往復テスト）

下流改変の完了判定は次の 1 本の往復テストで行う。AnalogBoard・gcsa・sys_app の三者を貫く。

  1. AnalogBoard が計測 Zarr ストアを書く（pulse_features は記録済み権威）。
  2. gcsa がストアを開き pulse_features を無改変・無再計算で返す（G2）。
  3. sys_app がストアを取り込み、24 名前付き列・N 行の Parquet と GMI JSON へ射影（S1〜S5）。extract_pulse_features は呼ばれない。
  4. sys_app で scatter＋gate 統計クエリを実行し、値がストアの pulse_features と（選定精度で）一致。
  5. （gate 出力する場合）gate を AnalogBoard→gcsa→sys_app→戻しで往復し、型/軸/座標/色が保存される（Part C）。
合格基準

再計算スパイが 0 回、Parquet が正名 24 列・N 行、scatter/gate 値がストア一致、gate 往復が無損失。これが満たされれば「gcsa と sys_app でシームレス」を担保する。
