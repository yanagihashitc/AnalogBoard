概要・実行前提

暫定・書き換え前提（必読）

本ドキュメントは暫定であり、確定版ではない。以下を前提とする：

  • 実行タイミング：この改変は AnalogBoard 側の改変（新アプリの出力実装）が終わった時点で実行する。それまでは着手しない。
  • 書き換えが起きる：AnalogBoard 側の実装確定に伴い、出力契約（Zarr スキーマ、meta.json フィールド、特徴量の順序/名前、精度、gate 形式）はまだ変わりうる。したがって本プランの各項目（特に file:line・フィールド名・型）は実行時に必ず再検証する。
  • スナップショット：本プランの sys_app/gcsa への参照（file:line・関数名）は現時点のソースに基づくスナップショット。実行前に該当箇所を再確認すること。
  • 直接改変しない：プロジェクトルールにより gcsa/sys_app は本作業では直接編集せず、本書は改変プロンプトとして提示する。実行は各リポジトリの担当が行う。

AnalogBoard 再構築（親プラン D5/D9/D10/D11/D19/D21/D22）で、本アプリは A/H/W を算出してgcsaネイティブ Zarr ストア（測定配列チャンクは D19 で暗号化）に記録し、gcsa と sys_app の両方でシームレスに使えることを要件とする。gcsa は構造上ほぼシームレス（fl_waveform とマーカ意味論の追加が必要）。sys_app には(1) gcsa ストアのリーダが無い、(2) 取り込み時に A/H/W を無条件再計算する、という2ブロッカーがある。 解は ストア完全直読（D22・2026-07-10 転換）：import はストアの meta.json＋配列形状から Dataset 行を登録するだけ（registration-only・データ複製なし）とし、特徴量・波形の読取は gcsa 経由でストアを直接読む。（旧解「射影コピー」は、①ピーク RAM≒生データサイズで大データセット import 不可、②読者不在の全波形コピー、③D19 暗号化を平文コピーが無効化、の3問題により廃止——経緯は AnalogBoard/docs/260709-analogboard-sysapp-integration-study.html）

実行順序と依存関係

本改変は AnalogBoard 側の出力契約が凍結されてから着手する。sys_app（S）と gcsa（G）は契約凍結後に並行可能で、最後にデュアル・シームレス往復テストで締める。

依存

「固定入力契約」（次節）が凍結されるまで S/G は着手しない。凍結前に着手すると、契約変更のたびに手戻りが発生する。

Part A — sys_app 改変（直読版：S1,S3,S4,S6〜S11。S2/S5 は廃止）

各項目は改変プロンプト形式（what / why / how / verify）。file:line は現ソースのスナップショット、実行時に再検証する。2026-07-10 改訂（D22）：S1 を「射影コピー」から「registration-only import＋直読リーダ」へ再設計し、射影 ETL 前提だった S2/S5 を廃止（不変条件は直読リーダに内在化）、S9〜S11 を追加した。実装詳細は sys_app リポジトリの実装プラン（sys_app/docs/plans/260710-analogboard-store-and-desktop-plan.html）に展開する。

S1. registration-only ストア取り込み＋直読リーダ（旧：射影 → 2026-07-10 再設計）

What(a) registration-only import：ストアの meta.json＋配列形状から Dataset 行（event_count・channels_json・store パス）を登録するだけにする——データ複製なし・status=finalized のみ対象。(b) 直読リーダ：特徴量読取を DuckDBFeatureStore（Parquet）から gcsa 経由の ZarrFeatureStore（復号・パーティション連結・event_idx 合成、返り値 dict[str, np.ndarray] 互換）へ差替える。bin 取り込みは互換で残す。
WhyD22。旧射影は①全計測 RAM ロード（ピーク RAM≒生データサイズ、1M イベント≒62GB で import 不可）②読者不在の全波形コピー（data.zarr、dataset_import_steps.py:787-822 に書くが読むコード無し）③D19 暗号化を平文コピーが無効化、の3問題。sys_app の特徴量読取は DuckDBFeatureStore 1箇所（duckdb_feature_store.py:100-148・呼出毎 in-memory 接続・read_parquet(path)）にチョークしており、コンシューマは gate（gate_mask_operations_mixin.py:112）と可視化（visualization_pulse_event_source.py:375）の2モジュールのみ——同型 dict を返せば上位は無改修。
Howimport：import_from_path で store 判定（.gcsa_store.json/meta.json 有）→登録専用ステップへ分岐（既存 build_importing_dataset/run_pipeline_in_transaction/メタ読取パターン dataset_import_persistence_service.py:271-284 を再利用）。リクエストスキーマはストアパス指定・スキャン/発見はストアベース・idempotency はストア指紋（meta.json の dataset id＋write_generation）。channels_json のレンジは meta の min/max（D21）から取得（全行スキャン不要）。display_name/comment/tags は表示名に採用。読取：DuckDBFeatureStore と同一シグネチャの ZarrFeatureStore を実装し2箇所の生成部で差替え（gcsa の get_pulse_features(features=[...])・列指定読取を使用）。extract_pulse_features は一切呼ばない。
Verifystore のみのディレクトリの import が秒未満で完了し Dataset 行が正しい（event_count==N・channels_json レンジ==meta の min/max）。PulseFeatureExtractor.extract_all 不呼出（spy）。scatter/gate クエリ値がストアの pulse_features と bit 一致。status=open のストアは import 拒否（明示エラー）。バルクコピー（parquet/data.zarr/waveforms.json）が生成されないこと。

S2. 廃止（2026-07-10・直読転換に伴い解消）

廃止の経緯

旧 S2「特徴量算出を記録済みが無い時のみに条件化」は、import が何もデコードしなくなる registration-only 化（S1）で対象コードパス自体が消えるため廃止。不変条件（D5：記録済み A/H/W を黙って上書きしない）は直読リーダ（S1b）に内在化され、受け入れテストの再計算スパイ（0回）で検証する。レガシー bin import 経路の無条件再計算（dataset_import_steps.py:239）は互換経路として残るが、ストア経路とは独立。gcsa 側の同じ不変条件は G2 が担う。

S3. channels_json の名前・順序・レンジを meta.json から決める（直読版に改訂）

Whatチャンネル名と順序を meta.json の宣言リストから、値レンジを meta.json の min/max（D21）から決める（dict 挿入順・全行スキャンに依存しない）。
Why位置行列(N,24)に名前が無く、現状は dict キー順から推定（dataset_import_steps.py:286-302）。誤ラベルは gate 軸検証（GateAxisMismatchError）と統計参照（routers/gates.py:298-315）を壊す。レンジは現状 import 時の全行スキャンで計算しており、registration-only 化とは両立しない——D21 の min/max メタが代替。
Howmeta の順序名を読み、列数と一致をアサートし、build_pulse_feature_channels（persistence 81-98,170-207）へ名前＋min/max を渡す。ストア宣言名 == channels_json となる検証を追加。
Verifychannels_json が meta の順序・レンジと一致。FSC_A/SSC_H の gate が解決し統計が出る。登録時に pulse_features 本体を読まない（I/O 監視）。

S4. GMI 波形の直読化（旧：waveforms.json 生成 → 2026-07-10 再設計）

What波形表示・export をストアの gmi_waveform(N,5,2400) の直接スライス（gcsa get_gmi_waveform(event_indices=...)）に切替え、import 時の waveforms.json 生成を廃止。プレビュー上限（GMI_WAVEFORM_MAX_COUNT=128、dataset_import_steps.py:47）は維持。
Why現行 waveforms.json は 128 イベント上限の JSON プレビューで、生成には全計測ロードが要る（registration-only と非両立）。読取チェーンは1本（resolve_waveform_source、visualization_waveform_source.py:193-220）・コンシューマ2箇所（波形エンドポイント・export）で差替えは小さい。誤順は GMI プロットを黙って取り違える。
Howresolve_waveform_source にストア直読分岐を追加し、取得した配列を既存 build_multi_channel_waveform_payload（waveform_payload.py:141-215・array-like を受ける）へ渡す。resolve_waveform_store_dir の dataset.zarr_path 依存（visualization_waveform_helpers.py:26-34）をストアパス項目に置換。meta の GMI 順 == FH_CHANNEL_NAMES をアサート。
Verify波形エンドポイントが 5 ch を返し、既知イベントの各サンプルがストアの gmi_waveform と一致（復号込み）。import 後に waveforms.json が存在しない。

S5. 廃止（2026-07-10・registration-only 化に伴い解消）

廃止の経緯

旧 S5「ストアモードでの入力検証の緩和」（cfg/bin 任意化・byte 検査スキップ・event_count をストアから導出）は、ストア経路が bin パイプラインを通らなくなる S1（registration-only）に吸収され独立項目として消滅。event_count はストア meta＋マニフェスト（D21）から取得し、pulse_features.shape[0] 依存による「取得中 import のサイレント切詰め」ハザードも status=finalized ガードで同時に解消される。レガシー bin import の検証は無変更で残す。

S6. 精度ポリシーの確定（直読版で適用範囲を更新）

備考2026-07-10：ストアデータセットの特徴量読取は直読（S1b・float64 のまま）になるため、本項の主対象は残る内部面——読取パスの round(6) 除去（直読にも効く）・レガシー bin import の Parquet・補償 sidecar（S9）の dtype——に更新。往復 bit 一致の判定基準は不変。
What確定（親プラン D15）：Parquet 列を float64 化し、読取の round(6) を除去する。
WhyD9 は float64、sys_app Parquet は float32（dataset_import_steps.py:488）＋読取 round(6)（visualization_service_helpers.py:47）で gcsa と bit 一致しない。
HowParquet 列を float64 化し round(value,6) を除去（float32＋許容差案は不採用）。
Verify抜取値がストアと選定精度で一致。読取が float coerce でエラー無し（duckdb_feature_store.py:67）。

S9. 補償（compensation）列の sidecar 化【新規 2026-07-10・唯一の設計変更】

WhatComp- 補償列を、特徴量ファイルへの上書き追記から sys_app 内部の sidecar ストア＋読取時の合成ビューへ分離する。
Why現行 CompensationService は特徴量 parquet を pyarrow で読み、Comp- 列を追記して同じファイルを原子的に書き直す（compensation/service.py:278-333,507・parquet_io.py:138-204・除去は列 drop＋rewrite）。読み取り専用の暗号化ストア（D19/D21 単一書き手規則）とは構造的に両立しない。直読転換で唯一「リーフ差替えでなく設計変更」が要る箇所。
How補償列を (dataset_id, feature_schema_version, write_generation, 行列ハッシュ) キーの sidecar（parquet 等・{source_dir}/sys_app/store/ 配下）へ書き、ZarrFeatureStore（S1b）と補償自身の読取パスで「ストア直読列＋sidecar 列」の合成 dict を返す。適用/再適用/除去のライフサイクルは sidecar の生成/差替え/削除として再実装。補償対象が _A/_H のみである既存規約は維持。
Verify補償適用→scatter/統計に Comp- 列が現れ、除去→消える（ストア本体のファイルは一切変更されない——mtime/ハッシュ監視）。補償値が旧実装と数値一致。write_generation が変わった（再アーム追記等）データセットで sidecar が無効化される。

S10. CI：凍結ストアコーパスの直読回帰テスト【新規 2026-07-10・オーナー決定】

WhatAnalogBoard 産の凍結ストア（フェーズ0 往復プロトタイプ産で bootstrap、実機データで拡充）を CI に常設し、「登録→直読→scatter/gate 統計がゴールデン一致・再計算スパイ0回」を回帰させる。

VerifyCI で常時緑。ゴールデン更新は契約変更 PR とセットでのみ許可（レビュー規約）。

S11. 旧射影機構の退役【新規 2026-07-10】

Part B — gcsa 改変（G1〜G5・2026-07-10 拡大）

G1. 正準ストアスキーマの凍結・公開＋可視性マーカ意味論（2026-07-10 拡張）

WhatAnalogBoard が書き sys_app が直読する正準ストアスキーマを凍結し公開する。拡張（D21）：可視性マーカの additive 字段（status/finalized_at/write_generation/パーティションマニフェスト/特徴量毎 min/max）を許容し、リーダ意味論を定義する——status=open のデータセットは既定で読取拒否（明示エラー）、opt-in で sealed prefix のみ読む。
Why両側が同じ契約を参照する単一の真ソースが必要（順序名・GMI名順・dtype/shape）。gcsa の従来慣習（meta.json 最終書込＝暗黙のコミット・pipeline/measurement_streaming.py:136-138 の save_meta）は AnalogBoard の再アーム append（meta.json を先に書く）と両立しない——明示マーカが代替。要注意（2026-07-10 検証）：gcsa の DatasetMeta.from_dict/to_dict は未知トップレベルキーを round-trip で黙って落とす——「additive だから無視されるだけ」は成立せず、gcsa が save_meta を呼ぶ経路でマーカ字段が消失する。
How版付きスキーマ文書＋、正準の順序特徴量名（== PulseFeatureExtractor.extract_all の 8ch×{_A,_H,_W}）・GMI 名順（== FH_CHANNEL_NAMES）・配列 dtype/shape・min/max を返す小 API を追加。マーカ字段はDatasetMeta の first-class フィールドとしてモデル化（未知キー透過に頼らない）したうえで意味論を実装。
VerifyAPI 出力が実 extract_all の並びと一致。マーカ字段付きストアが既存フローで無改変ロードし、gcsa 経由の meta round-trip（load→save）で字段が保存される。status=open の全読みが明示エラー。文書の版が meta.json versions と対応。

G2. ストア読取で記録済み特徴量を権威にする（無再計算）

Whatストア読取・解析が記録済み pulse_features を権威として返し、既存時に黙って再計算しない。
WhyD5：アプリ算出が権威。gcsa 側の再計算が既定だと記録値と食い違う恐れ。
Howストアリーダは格納 pulse_features をそのまま返し、再計算は明示 opt-in にする。
Verifyずらした値を含むストアを開き、解析が記録値を返す（再計算しない）。

G3. fl_waveform 配列型の追加（必須昇格）＋ gate モデル整合

Whatfl_waveform(uint16 (N,8,2400)) を ZarrStore に追加（gmi_waveform 対称・read/write 両方）。加えて gate モデル整合（histogram-range、all/root、余剰キー許容、色正規化）。2026-07-10 更新：ZarrStore に fl_waveform は未実装と確認済み（store 配下 0 hit・CSV エクスポータのみ）。D10 既定ON＋直読の前提として契約凍結前必須へ昇格。
WhyD10：生FL保全でアルゴリズム改訂時に再計算可能に。現状 ZarrStore は算出特徴量＋生FH のみ保持。gate 型集合が sys_app と非一致。
How_gmi_waveform_path/get_gmi_waveform/append_gmi_waveform を _fl_waveform_path/get_fl_waveform/append_fl_waveform として鏡像化（無ければ従来挙動）。chunk (2000,None,None)。BaseStore 抽象メソッドと MemoryStore のパリティも揃える。gate は histogram-range 受理 or range/threshold への合意下map、all/root の対応、色正規化を合意。
Verifyfl_waveform 有りのストアが既存フローで無改変ロード、get_fl_waveform が (N,8,2400) uint16・FL 順 FSC,SSC,FL1..FL6 で往復。

G4. AES-256-GCM チャンク暗号コーデック【新規 2026-07-10・D19・契約凍結前必須】

What独自 numcodecs コーデック（ID 例 gcsa-aes・Zarr v2 用）を実装・登録する：Blosc 圧縮済みチャンクの AES-256-GCM 暗号化/復号、チャンクヘッダに key_id（1バイト）、難読化した共通鍵の埋込。
WhyD19。AnalogBoard の C++ ライタが書く暗号化チャンクを gcsa（＝sys_app の直読層）が復号できる必要がある。抑止レベル（バイナリ解析には耐えない）を明記した設計。GCM 認証タグはチャンク単位の破損検知を兼ねる。
Hownumcodecs Codec 実装＋numcodecs.register_codec を import gcsa 時に自動実行。書込側の選択は ZarrStoreConfig(compressor=...) のプログラム的注入（GCSA_COMPRESSOR env allowlist {zstd,lz4} は変更しない）。読取側は .zarray の codec ID からレジストリ解決されるため ZarrStore 読取路の変更は不要。鍵は難読化定数＋key_id 引きの key provider に隔離（将来の鍵替え・外部鍵化の差替え点）。C++ ライタ側とワイヤ形式（nonce/tag/key_id の配置）を仕様書で固定し、フェーズ0往復の合格条件とする。
VerifyC++ ライタ産の暗号化ストアが import gcsa 済み環境で無改変ロード（フェーズ0往復）。改竄チャンクが GCM 認証エラーで fail-loud。コーデック未登録環境では「未知コーデック」の明示エラー（黙って生バイトを返さない）。

G5. リーダ堅牢化：バリデータ・並行性・CI コーパス【新規 2026-07-10】

What(a) パーティションバリデータをフェーズ0のシャーディング決定に追従させ（2026-07-10 検証：round-robin の数式はバリデータだけでなく index-gather 経路（_zarr_artifact.py:415-427）と interleave 逆写像にも焼き込まれている——変更時は読取経路全体が対象）、スライス読みの長さ検証バイパス（サイレント不整合の窓）を D21 マニフェスト参照で閉じる。(b) 読取並行性：web サーバ利用（sys_app 直読）向けに multi-worker 指針を文書化し、必要なら per-instance RLock（全読取を直列化・解凍含む）の粒度を縮小。docstring「Cross-process access」と実態（プロセス内ロックのみ）の不一致を解消。(c) 凍結ストアコーパスの読取回帰を gcsa CI に常設（sys_app S10 と同一コーパス共有）。
Why直読転換で gcsa の読取路が sys_app の権威読取面になるため、途中状態の誤読・並行読取の直列化ボトルネック・契約回帰が直接ユーザ影響になる。
Howバリデータはマニフェスト（配列毎行数・sealed）を第一情報源に。ロック粒度変更はキャッシュ不変条件（stat シグネチャ）に注意し、当面は「プロセス毎 ZarrStore インスタンス＋multi-worker」の運用指針でも可。CI は pytest＋コーパス参照。
Verifyシャーディング決定後のストアで全読み/スライス読みが正しく検証される。並行読取ベンチで直列化ボトルネックが許容内。CI コーパス緑。

gate は「取得データ」とは別の合意事項。gcsa v1 と sys_app で gate 型集合が異なるため、往復（AnalogBoard→gcsa→sys_app→戻し）が壊れない対応表を合意する。

受け入れ（デュアル・シームレス往復テスト——直読版）

下流改変の完了判定は次の 1 本の往復テストで行う。AnalogBoard・gcsa・sys_app の三者を貫く。旧版の「射影忠実性」の脚は直読転換で消滅し、テストは簡素化された。

  1. AnalogBoard が計測 Zarr ストアを書く（pulse_features は記録済み権威・チャンクは D19 暗号化・finalize 済み＝status=finalized）。
  2. gcsa（G4 コーデック登録済み）がストアを開き pulse_features を無改変・無再計算で復号して返す（G1/G2/G4）。
  3. sys_app がストアを registration-only で登録し（S1a・複製なし・秒未満）、scatter＋gate 統計クエリを直読（S1b/S4）で実行——値がストアの pulse_features と bit 一致。extract_pulse_features は呼ばれない。
  4. （gate 出力する場合）gate を AnalogBoard→gcsa→sys_app→戻しで往復し、型/軸/座標/色が保存される（Part C）。
  5. 補償（S9）適用→除去がストア本体を変更しない（mtime/ハッシュ不変）。
合格基準

再計算スパイ 0 回、登録がデータ複製ゼロ・秒未満、scatter/gate 値がストアと bit 一致、status=open の import が明示拒否、gate 往復が無損失、ストア本体が下流から一切書き換えられない。これが満たされれば「gcsa と sys_app でシームレス」を担保する。本テストは S10/G5 により CI コーパスとして常設化する。
