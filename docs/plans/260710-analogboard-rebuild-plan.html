エグゼクティブサマリ

現行 AnalogBoard は Windows の MFC ダイアログアプリ（AnalogBoard_TestApp）＋ Cypress CyAPI をラップする DLL（AnalogBoard_Dll）で、FX3 USB3 経由で FPGA ボードから高速多パラメータ計測データを取得する。本プランは、オーナーの7要求（直感的UI・新ドライバ対応・堅牢化・リアルタイム scatter/histogram/gating・4GB制限撤廃・自動再接続・診断ログ）を満たすための再構築方針を定める。

2件の多エージェント調査（コード・FPGA仕様・git履歴・現場ログの横断精査）と、オーナーからの重要な追加情報（後述の「真因」）を突き合わせた結論は「ストラングラー型リビルド」である。initial commit へ戻す全面やり直しは推奨しない — それは現場で鍛えられた実証済み資産（テスト済みポリシー群・レジスタ意味論・SimRunner）を捨て、god class と欠陥DLLの骨格へ後退することを意味する。代わりに、UI・描画・オーケストレーションは新スタックで作り直し、取得・通信コアはネイティブ C++ の新設計としつつ dev ブランチの実証資産を移植する。UIスタックの推奨は C#/.NET 8 WPF ＋ ネイティブ C++ コア（C ABI）。

最大の技術リスクだった「18回の現場イテレーションでも閉じなかった高密度EP6タイムアウト」は、オーナーの調査により物理層（機械側ケーブル先端のインラインon/off切替スイッチ）が真因と判明し、これを外すことで Win10/Win11 とも高密度で96ウェル1枚を完走できるレベルまで解消した。この判明により最大リスクはブロッカーから大幅に格下げされ、通信コアの再設計は「未診断問題の引き継ぎ」ではなく「堅牢化・自動再接続・4GB再アームのための設計改善」に純化する。加えて実機アクセスが希少という制約に対し、実機通信を録画して決定論的に再生するエミュレータ（Tier 1〜3、2枚目FX3をデバイスエミュレータとする HIL を初期から含む）を一級ワークストリームとして構築し、日常開発・回帰・UI/描画開発を実機ゼロで回せるようにする。

D19ストア暗号化チャンク暗号化を採用（抑止レベル）。AEAD は Store 層で実施（2026-07-10 方式確定）測定配列チャンクのみ Blosc(lz4)圧縮→AES-256-GCM で保護。鍵はアプリ（C++コア）と gcsa が共有する埋込鍵、チャンクヘッダの key_id（1バイト）で将来の鍵替えに備える。抑止レベル（配布物内に鍵が存在する以上、解析には耐えない）と明記した上での採用。meta.json・.gcsa_store.json・gate（正準v1）は平文維持。実装層（2026-07-10 オーナー判断・codex レビュー起点）：numcodecs の Codec は encode(buf)/decode(buf) のみでチャンク座標を受け取らない（numcodecs 0.13.1 abc.py・zarr 2.18.3 core.py で実証）。よって当初案の「カスタム numcodecs コーデック」では座標を AAD に束縛できず、読取側の AEAD は zarr の Store ラッパ層（__getitem__ がチャンクキーを受け取る）で行い、.zarray の compressor には内側の Blosc を宣言する。zarr v2 の DirectoryStore ではチャンクキー＝ファイル名のため、C++ ライタ（直接ファイル書込）と gcsa の Store ラッパは同一の AAD を計算でき相互運用が成立する。帰結：c-blosc＋最小自作ライタが実質第一候補（D12 のフォールバック昇格）、フェーズ0往復検証は保護込み、gcsa へ G4（Store ラッパ＋鍵管理）。GCM 認証タグによりチャンク単位のサイレント破損検知が付随する。同一座標への旧チャンク差し戻しは AAD では防げないため D21 マニフェスト（write_generation）で担保する。詳細は「リスクと未決事項」節の提案 P1 参照。2026-07-09 採用／2026-07-10 実装層を確定。
D20アプリ統合方針AnalogBoard と sys_app は別アプリ維持コードベース統合はしない（3視点独立調査で一致・docs/260709-analogboard-sysapp-integration-study.html）。到達形は「一製品・複数プロセス」——同梱配布＋自動ハンドオフ（近期）、必要なら WPF シェルへの WebView2 埋込（将来アドオン）。根拠：sys_app は ML 追加で頻繁更新（リリースサイクル非対称）、取得側は常駐ネイティブ必須（FR-11）＋障害ドメイン隔離、解析数値核は gcsa（Python+Rust）。sys_app は ML/DR 実装後に Tauri デスクトップ化を計画。2026-07-09 決定。
D21ストア可視性契約finalization マーカ＋パーティションマニフェスト導入meta.json に additive：status: open|finalized・finalized_at・単調増加 write_generation・パーティションマニフェスト（配列毎行数＋per-partition sealed）・特徴量毎 min/max レンジ。単一書き手規則（datasets/<id>/ の書き手は取得コアのみ）＋「ストア＋meta.json＝唯一のクロスアプリレジストリ」。取得中 import のサイレント切詰め防止・再アーム append と gcsa の meta.json-last 慣習の非両立を解消・クラッシュ回復に単独価値。出力契約凍結前に確定、フェーズ0往復で gcsa 許容を検証。2026-07-09 決定。
D22sys_app 読取方式ストア完全直読（registration-only import）射影コピー（旧 S1）を廃し、sys_app は登録のみの import＋gcsa 経由のストア直読へ。根拠（実コード調査 2026-07-10）：旧射影はピーク RAM≒生データサイズで大データセット不可・読者不在の全波形コピー・D19 暗号化を平文コピーが無効化。読取 seam は狭く（DuckDBFeatureStore 1箇所・波形チェーン1本）差替えは S/M、唯一の設計変更は補償列の sidecar 化。D11 の達成手段変更であり要件は不変。2026-07-10 決定。

キーコンセプト

ストラングラー型リビルド
既存アプリを一度に捨てず、実証済みコア（通信/エンジン/ポリシー）を残しつつ新UI・新機能で段階的に置換する方式。全面リライトより現場検証面が小さく、各段を独立に実機ゲートできる。
ネイティブコア ＋ C ABI
USB取得・デコード・ファイル出力など時間敏感な処理を C++ に閉じ、UI（C#）とは C ABI 越しに疎結合。生バイト列はネイティブから出さず、集計済みデータのみ受け渡す。
録画→再生エミュレーション
実機セッションで EP2/EP4/EP6 の生通信・タイミング・configを録画し、後で決定論的に再生する仕組み。障害トレースも再生でき、実機なしで開発・回帰・障害対応が回る。
HIL（2枚目FX3）
Hardware-In-the-Loop。もう1枚の FX3 をデバイス側エミュレータとして走らせ、録画データを実 USB・実 cyusb3.sys 経由で再生する。ソフト単体では届かない転送タイミングやドライバ経路を検証する。
gating（ゲーティング）
scatter/histogram 上で範囲（矩形・ポリゴン等）を選択し、条件を満たすイベントだけに絞り込む操作。本アプリの gate 定義は下流 ../sys_app / gcsa と互換フォーマットで入出力する。
4GB再アーム
FPGAのDDR書込アドレスが非リング（4GiBで停止）のため、取得しながらドレインし、満杯前に停止→再INITで書込アドレスをリセットして取得を継続する方式。破棄を再アーム窓に限定する。
DFX / Directed Power
Win11 の新ドライバ INF が有効化する WDF のアイドルサスペンド機構。新ドライバでの通信不可の最有力仮説。NoDfx レジストリA/Bで検証する。
Type A/B/C 障害分類
現場ログの障害分類。A=取得中のEP6タイムアウト（両ENDビット0）、B=ドレイン後のタイムアウト（WR_END=1/RD_END=0）、C=正常完了。新ログでもこの語彙を踏襲する。

方針判断（改修 vs 作り直し）

調査の結論は「作り直し寄り、ただし通信コアは dev の実証資産を活かすストラングラー型」。判断の根拠は以下。

論点事実（調査より）示唆
目玉機能の基盤 scatter/plot/histogram/gating・サンプルデコード・スレッド安全な producer-consumer は main に存在しない（raw EP6 を memcpy で .bin へ直行）。dev には BlockingQueue＋エンジンあり。 描画・デコード・データチャネルは改修でも作り直しでも新規。UIを残す利点が小さい。
UIの構造 Dialog1_Main.cpp は約3,927行の god class（全コードの約47%）。ゲイン用に13ch×5段の65個のコンボボックス、固定694×339レイアウトに約230個の絶対配置コントロール、ボタンのキャプション文字列を状態機械に流用。 目標UX（1スライダー、リアルタイム描画）とは非連続。UIは全面新規が妥当。
改修の実績 過去のin-place改修は3回現場で退行（0.1.5/0.1.6アロケータ変更、0.2.0エンジン抽出の空キャプチャ、ホットパスlogで取得タイミング破壊）。いずれも単体テストは緑だった。 「テスト緑」は弱い証拠。受け入れはOSごとの実機ゲートが必須。
移植可能な資産 MFC非依存のヘッダオンリー・ポリシー群 約1,871行（22%）＋単体テスト約4,403行、レジスタ意味論ドキュメント、SimRunner、.bin ゴールデンテスト。 作り直しは「ゼロから」ではなく「仕様＋回帰資産つき」で始められる。
DLLの状態 約542行のMFC拡張DLL。C++名前マングリング export、ファイルスコープのシングルトン状態、転送ごとのハンドルリーク、_msize() のCRTまたぎUB、EP6サイズ検査のコメントアウト、単一グローバルmutexでEP2/EP4/EP6を直列化。 自動再接続・poll-while-streaming・堅牢化のいずれにも再設計が必要。
真因判明による更新

「作り直しが未診断のタイムアウト問題を引き継ぐ」という最大の反対理由は、真因が物理層（切替スイッチ）と判明したことで解消。結果として、通信コアはフルリライトではなく dev の現場検証済みコアを活かして堅牢化するストラングラー型が相対的に有利になった。ただし堅牢化・自動再接続・4GB再アームのための通信層手術は要求由来で依然必要。

タイムアウト問題の真因

オーナーの調査で、機械側から出ているケーブルの先端に付いていたインラインの USB on/off 切替スイッチ（メス—スイッチ—オスの区間）を外したところ、Win10/Win11 とも高密度で96ウェル1枚分の計測をエラーなく完走できるレベルまでタイムアウト率が低下した。USB3 SuperSpeed の差動ペア上にある機械式スイッチは、インピーダンス不整合・スタブ反射・接点抵抗を持ち込み、転送レートが上がるほどリンクエラー→リカバリ→バルク転送ストール（＝タイムアウト）を誘発する。

これまで「説明がつかない」とされた手がかりが、すべてこの一点で整合する：

これまでの手がかり物理層（切替SW）での説明
低密度は常にクリーン、高密度でだけタイムアウトマージナルなリンクは転送レートが上がると破綻する（典型的なUSB3信号品質症状）
同一バイナリ(0.1.4r7)がWin10クリーン・Win11タイムアウト別マシン＝別ケーブル/ポート経路。OS差と思われた一部はリンク品質差
18回のホスト側イテレーションでも閉じないそもそもソフトの問題ではなかった
タイムアウトに切断が伴う（Type A/B）物理層断の典型
「USBを抜き差しすると直る」マージナルな接点の再装着でリンクが張り直される
設計への最大の教訓

本質は「物理層の問題をソフトが検出・可視化できなかったため、何ヶ月もソフトを疑い続けた」こと。オーナーの「エラー原因を究明できるように」という要求は、抽象的なログ機能ではなく USBリンク健全性の可視化（ネゴシエート済みリンク速度の検出＋SuperSpeed/USB2フォールバック警告＋タイムアウト/リトライ/切断カウンタ＋「インラインスイッチ等の物理層を確認」ガイド）として実装する。これがあれば次に同じ事象が起きても即座に配線問題と切り分けられる。

別問題として残る軸

Win11 の新ドライバでの通信不可（ロールバックで回復）は信号品質では説明できない別問題であり、DFX/SDK世代不一致の線として引き続き検証する（後述）。また「ほとんど出ない」はゼロではなく、接点は劣化・個体差もあるため、直結（良好な物理接続）前提の上でソフト側の自動再接続・リトライ＋再同期は要件のまま残す。

ターゲットアーキテクチャ

時間敏感な取得パスは C++ に閉じ、UIは C#。両者は C ABI で疎結合。生バイト列（〜200MB/s）はネイティブから出さず、UIへ渡すのは集計済み（ビニング済みscatter・ヒストグラム・カウンタ・固定上限のサンプリング済みGMI波形スナップショット＝FR-13）のみ。エラー処理・録画・ロギングはコア側に一元化する。

設計の要点

  • プロセス/スレッド分離：取得の読取スレッド、デコード、ファイル書込を分離し、BlockingQueueで背圧をかける。UIのハングやログ出力が取得ホットパスを乱さない（過去の「計装が取得を壊した」失敗を構造的に防ぐ）。必要なら取得コアを別プロセス化してUIハングと完全隔離。背圧の定量規定（2026-07-10 追加）：キュー容量はバイト上限＋high/low watermark で定義し、満杯時の動作＝EP6 読取スレッドをブロック（DDR がバッファするため安全。drop はしない）＋UX-11 警告＋必要なら先行再アームのトリガ。数値はフェーズ1で確定し Tier1/2 で検証する。
  • 生データは越境させない：C ABI を渡るのは密度ビニング済みscatter・ヒストグラム・カウンタ・イベント特徴量、および固定上限のサンプリング済みGMI波形スナップショット（FR-13。既定〜100波形×2400サンプル、コア側でサンプリング／必要ならデシメーション済み——生ストリームの転送ではない）のみ。GCが200MB/sの経路に触れない設計ルールを初日から強制（WPF採用時）。
  • 通信層の刷新点：エンドポイント毎の明示的タイムアウト、Abort()/Reset()→ReConnect()による回復、BeginDataXfer/FinishDataXferのキュー型非同期転送、EP6バルク読取に直列化されない独立したEP4ステータスチャネル、UsbdStatus/NtStatus/LastErrorのログ化。
  • C ABI 中立性ルール（2026-07-10 追加）：C ABI を越えるのは版付き POD／シリアライズ可能構造体のみ。コマンド（start/stop・一括設定適用・レシピ読込）とフィードフレーム（ビニング済み scatter・histogram・カウンタ・GMI スナップショット ≤100×2400・リンクヘルス）を列挙型で定義し、.NET/WPF/COM 型・コールバックのオブジェクトグラフを禁止。コールバックは frame-ready 通知＋pull か polled event queue に限定。これにより同一コアを in-proc（WPF・フェーズ1〜4）／常駐サービス／将来の別フロントエンドから無改修でホスト可能にする。フェーズ1のテスト計画に非WPF最小ホスト（素の C/C++ or コンソール）で start→取得（エミュ）→再アーム→stop→ストア書込の全周を駆動する Tier1 適合テストを置き、規則をコンパイル／テストで機械的に強制する。
  • フィード fan-out ポイント（2026-07-10 追加）：集計フレームは BlockingQueue の消費側（デコード/集計スレッド。EP6 読取・ZarrWriter スレッドの外）で生成し、sequence 番号＋monotonic timestamp を付与する。multi-sink の attach point を設計上1箇所命名する（現在の sink は C ABI の1本のみ。第2 sink は実装しない——将来のミラー追加時にホットパスへ触れない構造だけを保証する）。sequence 番号は Tier1/2 回帰のフィード再生決定性にも使う。

sys_app シームレス契約

オーナー要件「本アプリで取得したデータは gcsa と sys_app の両方でシームレスに使える」を担保するため、sys_app（apps/api Python/FastAPI＋apps/web Next.js、ホストからコピー済み）の取り込み・特徴量・gate・可視化・保存契約を精読した。結論を先に述べる。

判定：gcsaは構造上シームレス、sys_appは「そのまま」では乗らない（2ブロッカー）

Zarr ストアは gcsa の形式そのものなので gcsa は改変ほぼ不要で読める。しかし sys_app には次の2点があり、追加対応が必須：

  1. sys_app に gcsa ストアのリーダが無い。sys_app の権威的な特徴量読取面は自前の sys_app/store/pulse_features_{id}.parquet（event_idx int64 ＋ 特徴量名ごとの float32 列）＋ GMI の waveforms.json。ZarrStore/gcsa_store の参照はゼロ。ストアだけ渡すと Parquet 不在で import が落ちる。
  2. sys_app は取り込み時に必ず A/H/W を生FLから再計算する（dataset_import_steps.py:239 で無条件に extract_pulse_features）。＝ D5（アプリ算出を権威）が無視・上書きされる。

朗報：sys_app の読取・可視化・gating 層は既に D5 と整合的（記録済み Parquet を優先し、再計算はフォールバック）で、座標は plotted 値も gate 幾何も生ADC線形で gcsa と一致（log/biexp は表示時のみ）。当初は「取り込み時の射影（projection）コピー」を解としたが、2026-07-10 のオーナー決定（D22）で「registration-only import＋ストア完全直読」へ転換した（下記）。

直読（direct read）アーキテクチャ（2026-07-10 改訂・旧：射影）

sys_app は取り込み時にデータを複製せず、ストアの meta.json＋配列形状から Dataset 行を登録するだけ（registration-only import・ミリ秒級）とし、特徴量・波形の読取は gcsa 経由でストアを直接読む。転換の根拠（2026-07-10 実コード調査で確定）：

  • 旧射影パイプラインは全計測を RAM にロードし（ピーク RAM ≒ 生データサイズ。1M イベント ≒ 62GB）、大データセットの import が構造的に不可能
  • 読者が存在しない全波形コピー（data.zarr・~62KB/イベント）を書いていた（純粋な無駄）
  • D19 暗号化と両立しない——射影は保護済みストアの隣に全データの平文コピーを作る
  • sys_app の読取 seam は狭い（特徴量は DuckDBFeatureStore 1箇所・コンシューマ2モジュール、波形は読取チェーン1本・128イベント上限プレビュー）ため、差替えコストは S/M。唯一の設計変更は補償（Comp- 列の parquet 上書き）の sidecar 化

実機エミュレーション戦略

エミュレータの価値は、パラメータから波形を合成することではなく、実機セッションで生の通信を丸ごと録画し、決定論的に再生することにある。特に失敗したセッションのトレース（タイムアウト・切断・ショートリードが起きた実ログ）を再生できると、ケーブルを抜き差しせずに自動再接続ロジックを開発・テストできる。録画は通信層に初日から組み込み、以後すべての実機セッションが自動的にコーパスを育てる。

録画データの取扱い（2026-07-10 追加・D19 との整合）

EP6 生録画はストアが暗号化する測定データそのもの（~200MB/s）であり、無制限の平文録画は D19 の保護を迂回し、ディスクを高速に消費する。規定：(1) 製品/現場ビルドの既定＝メタデータ・障害チャネル（EP2/EP4・タイミング・障害トレース）のみ常時ON、EP6 生ペイロード全録画は明示 opt-in＋直近Nの リングバッファ。(2) 容量上限とローテーションを設定化し、ディスク枯渇時は録画を先に止める（取得を止めない。UX-11 監視対象に録画を含める）。(3) 保存時保護はストアと同等（D19 コーデックの流用可）とし、コーパスの持出し（開発/CIへの搬送）は finalize 済み・匿名化判断を経たものに限る（運用の所有者はオーナー確認事項）。(4) recorder ON での 200MB/s soak をフェーズ1受け入れに含める（記録が取得を乱さない実証）。

録画コーパスに含めるもの

  • EP6 の生バイト列（実際の粒子イベントの波形データ）
  • EP4 のレジスタ推移（DDR_WR_END / DDR_RD_END / wave counter / FPGA_ST の時系列＝状態機械トレース）
  • EP2 コマンド列（再生側が正しく応答するため）
  • 転送間タイムスタンプ（レート・背圧の再現）
  • config タグ（gain/offset/サンプルレート）
  • 障害イベント（timeout/切断/short read の実トレース）

境界線（正直な整理）

エミュレータで置き換えられるのは機能・プロトコル・データの忠実度。置き換えられないのは物理層・実ドライバ層（信号品質タイムアウト＝例の切替スイッチ、Win11新ドライバ通信不可、実USBリンク再ネゴ、切断/抜き差し）。よってエミュレータは実機トリップの「頻度」を大きく減らすがゼロにはしない。実機は物理層・ドライバ検証・最終受け入れに絞って残す（次章）。

コーパスの三重価値

  • ①エミュレータの再生ソース
  • ②回帰・受け入れのゴールデンファイル（既存の.binバイト単位ゴールデン＋SHA256手順を踏襲）
  • ③scatter/gating開発用のリアルなイベント分布

既存の FpgaDdrModel（store-then-read DDR、RD_WAIT・stale-status忠実度）はTier 1のコアとしてそのまま移植する。

4GB制限の撤廃戦略

真因はFPGAのDDR3物理容量（4GiB）を、単調増加・非ラップの書込アドレスとラッチされる満杯フラグが強制していること。満杯時にサンプリングが停止し以後のイベントは破棄される。ホスト側だけでは破棄をゼロにはできず、4GiBごとの再アーム窓に限定できる。読み出しは書込領域を解放しない（＝ドレインしても1アームあたり4GiBの上限は不変）。

フェーズ1 — ホストのみ（作り直しに標準で組み込む）

  • 並行ドレイン：ハードは既に対応（BUFFER_CTRLがDDRをEP6 FIFOへ先読み）。連続ドレインで実効容量を延ばし、平均イベントレート < USBスループットの間はバックログを小さく保つ。
  • 先行サイクル（proactive cycling）：DDR満杯を待たず、約75%フィルで MEAS_ON をデアサートして停止。停止後の必須ドレイン時間とRD_WAIT固着の露出を両方bound。FW変更不要で最も安価なレバー。
  • 自動再アーム：DDR_RD_END後に MANUAL_MEAS_ON を0→1でパルスしてINITへ再突入、書込アドレスをリセット。現行auto-modeの外側ループが実証済みテンプレート。各再アームサイクルのイベントは同一データセットの Zarr パーティションへ appendする（D9）。番号Nのペアbin juggling は不要。
  • 互換 .bin 併用時のみの制約：移行互換で fl/fh .bin も出す場合は、同一N の FL/FH イベント数一致・bytes_per_cell厳密倍数・境界でヘッダ/パディング/バイトスワップ禁止を守る（gcsa互換契約参照）。主経路のZarr追記では不要。
  • 前提：新通信層のキュー型非同期EP6読取＋独立EP4ステータス＋writerスレッド分離（ディスク遅延がDDRを満杯へ押さないため）。
残留ロス

破棄は「停止→残バックログのドレイン＋約500us FPGA WAIT＋約10us INIT＋レジスタ往復」の再アーム窓に限定される。並行ドレインでバックログが小さければ窓は典型的に秒未満だが、ディスク停滞でバックログが膨れると最悪で数十秒になりうる。p99再アーム窓（ms）を実機ゲートの測定基準にする。

フェーズ2 — FPGA FW変更（真のゼロロスが必須の場合のみ）

  • 書込側をリングバッファ化（読取ポインタ追い越し保護）し、満杯強制停止を撤廃。
  • 同じFWリリースにabort/ソフトリセットレジスタを同梱（RD_WAIT固着＝電源サイクル必須の恒久解、自動再接続の上限も解消）。
  • 変更は約3 VHDLファイルに局所化、FPGAは約15%利用でリソース余裕、Quartus 21.1プロジェクト＋DDRモデルテストベンチはリポジトリ内。ただし現方針はFW非改変を宣言しており、ベンダ（PHINE Design）または内製Quartus体制の確保＋実機再検証が前提。

既知の上限

完全なホスト側再接続でも、FPGAがドレイン中にRD_WAITで固着した場合は回復不能（FWにabortレジスタが無い）。USB回復後は必ずドレインを完了する設計にし、恒久解はFWのabortレジスタ（4GB戦略フェーズ2）。また、FX3 FWがUSBサスペンド/LPMを扱わない場合、NoDfxは回避策でなく恒久的な出荷要件になる可能性がある。

再接続後のバイトストリーム再同期（2026-07-10 追加）：EP6 はフレーミング無しのフラットな生バイト列（固定 bytes_per_cell）のため、切断時に転送途中のイベントが発生しうる。規定：デコーダは bytes_per_cell 単位でストリームオフセットを追跡し、Abort/ReConnect 時は転送中バッファを破棄して次のイベント境界（bytes_per_cell 整列位置）から再開する。部分イベントは損失としてカウント（FR-04 の損失記録に計上）。切断位置を変えた障害注入（イベント境界上・中間・ヘッダ相当位置）を Tier2 回帰に追加する。

主要リスク

Mediumsys_app は Zarr ストアをそのまま読めない（直読転換で対応・2026-07-10 改訂）
sys_app に gcsa ストアのリーダが無く、取り込み時に A/H/W を無条件再計算する（D5を無視）。対策（D22）：registration-only import＋gcsa 経由のストア完全直読へ転換（旧「射影」は RAM 制約・平文コピー・読者不在コピーの3問題で廃止）。読取 seam は狭く（DuckDBFeatureStore 1箇所・波形チェーン1本）差替えは S/M、補償列の sidecar 化のみ設計変更。デュアル・シームレス往復テスト（直読版）をビルドゲートに。要件でありハード依存——放置すると「両方でシームレス」が満たせない。
MediumC++ からの Zarr v2 書き込み実装コスト（D19 で自作ライタが実質第一候補に）
主出力を gcsaネイティブ Zarr にするには C++ の Zarr v2＋Blosc＋AES-GCM 暗号化（D19）ライタが要る。acquire-zarr（D12 第一候補）はカスタム暗号コーデック段を挟めない可能性が高く、c-blosc＋最小自作ライタが実質第一候補に昇格。対策：フェーズ0で暗号化コーデック込みの往復プロトタイプ（G4 登録済み gcsa が実際にロードできること）を検証してから確定。ZarrWriterをモジュール隔離しEP6ホットパス外（writerスレッド）に置く。互換の生bin出力も残せるので最悪フォールバック可能。
Medium特徴量アルゴリズムが「仕様」になる（D5）
アプリ算出のA/H/Wを権威とするため、算出アルゴリズムが下流の真ソースになる。改訂時は再処理や版管理が必要で、算出は取得ホットパス外（decode/writerスレッド）に置く必要がある。対策：schema_version／meta.versions で版管理し、bring-up期は生FL波形を保全して再計算余地を残す。gcsa v1アルゴリズムを移植基準にしパリティを一度検証。

10万点超のライブscatter＋点ごとのgate着色＋lasso判定＋連動histogramを取得中に成立させる要件は既製チャートライブラリでは満たせない可能性。対策：スタック確定前に密度ビニングscatterの描画経路を合成10万イベントで先行プロトタイプ。ライブラリは「pro」でなく「risk」として扱う。
Lowgcsa/sys_app互換（契約抽出済み・残るは実行時確認）

検討済みの提案（クローズ）

採用 → D19 P1：Zarr ストアのチャンク暗号化——本アプリ・sys_app のみで開ける出力（2026-07-09 起案・同日採用）

動機：D9 の主出力（gcsaネイティブ Zarr）は標準の zarr ライブラリ・汎用ツールで誰でも開ける。出力データを本アプリと sys_app（gcsa 経由）のみで開けるようにしたい。

案の骨子：Zarr v2 のストア構造（partition 追記・atomic rename・自己記述 meta・4GB再アーム＝append）は D9 のまま変えず、配列チャンクのみ Blosc(lz4) 圧縮→AES-256-GCM 暗号化して書き、.zarray の compressor を独自 ID（例 gcsa-aes）にする。公開 numcodecs レジストリに存在しない ID のため、素の zarr / Python / 汎用ビューアでは「配列の形は見えるが中身は復号できない」状態になる。読取側は gcsa に numcodecs カスタムコーデックを登録する additive な小変更のみ（sys_app は gcsa 経由で読むため原則個別対応不要）。

  • 鍵管理＝抑止レベル：難読化した共通鍵を本アプリ（C++コア）と gcsa に埋め込む。チャンクヘッダに key_id（1バイト）を持たせ将来の鍵替えに備える。バイナリ解析には耐えない（アプリが開ける以上、鍵は配布物内に存在する）ことを明記した上での採用となる。解析耐性が必要ならライセンスサーバ／ドングル等の外部鍵権威が要るが、運用コストが大きく現時点では対象外。
  • 平文のまま残すもの：meta.json・.gcsa_store.json・gate（正準v1）。自己記述性・現場デバッグ性・サポート性を維持し、保護対象は測定配列本体（特徴量・波形）に限定する。
  • 副次効果：AES-GCM の認証タグでチャンク単位のサイレント破損を暗号学的に検知できる（「破損サイレント解消」への追加防御）。
  • 性能：暗号化は writer スレッド内（EP6 ホットパス外）。AES-NI で GB/s 級のため 200MB/s 経路に影響なし。圧縮→暗号化の順のため圧縮率も不変。

採用した場合の影響：① D12——acquire-zarr はカスタムコーデック段を挟めない可能性が高く、既承認フォールバックの c-blosc＋最小自作ライタが実質本命になる。フェーズ0 の gcsa 往復検証は暗号化込みで実施する。② gcsa——カスタムコーデック登録＋埋込鍵の追加を下流改変プランに G4 として追加（直接改変しない）。③ スコープ外——sys_app 取り込み後の内部 Parquet/DuckDB は平文のまま（保護対象は持ち出されるストアファイル。顧客環境内の常時保護まで求める場合は別途 S プロンプトが必要）。
