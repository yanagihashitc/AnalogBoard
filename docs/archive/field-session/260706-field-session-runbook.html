共通条件・判定語彙

全判定ランの固定条件

  • 機械からPCへ直結。SW-SEG-01、USBハブ、延長区間を入れない。
  • Manual Getは現場通常条件のOFF。機械側開始を使用する。
  • Release x64、デバッガ／DebugView／USBPcap／Wiresharkなし。判定中に取得ホットパスへ計装を追加しない。
  • 判定buildはtelemetry graceful-stop buildで固定し、同一EXE/DLLを新（N leg）／旧（B leg）driverで共用する。build ID・EXE/DLL SHA-256は「判定build・matched A/B総括」節と自己検証packageへpin済み。実機測定前に TELEMETRY_CSV_READY_1314\run_00_verify_package.batをPassさせる。旧判定build r7-driver-ep4-polling-20260715T1618JSTはsupersededとし、count・証跡は新buildで0から取り直す。
  • PC、driver version、DFX状態（dfx_status.ps1出力の保存）、EXE/DLL SHA-256、USB port/controller、サンプル、設定、保存先、NVMe、電源モードを各ブロックの開始前に記録する。
  • 成功・失敗を問わずログ、cfg、完成bin、.tmpを削除しない。

現地データ完全性集計（データ本体を移さない場合）

既存3runでは実施済み

260714_1611／1613／1616はworkspaceコピーを直接監査済み。以下のcollectorは、今後データ本体を持ち帰れないrunに使用する。

次の非破壊PowerShellをCollect-FieldEvidence-v1.ps1として保存し、runごとに実行する。出力フォルダだけを持ち帰る。スクリプト本文、版名、実行コマンド、標準出力も証跡に含める。

$CollectorVersion = "FieldEvidence-v1"
$RunId = "260714_1611"
$DataDir = "D:\Analyzer_data\260714"
$OutDir = Join-Path $DataDir ("evidence_" + $RunId)
New-Item -ItemType Directory -Force -Path $OutDir | Out-Null

$files = @(Get-ChildItem -LiteralPath $DataDir -File |
  Where-Object { $_.Name -like ($RunId + "*") })
$files | Select-Object Name,Length,LastWriteTimeUtc |
  Export-Csv (Join-Path $OutDir "file_inventory.csv") -NoTypeInformation -Encoding UTF8

$parts = @($files | ForEach-Object {
  if ($_.Name -match '_(fl|fh)_(\d+)\.bin$') {
    [pscustomobject]@{ Stream=$Matches[1]; Index=[int]$Matches[2]; Length=$_.Length; FullName=$_.FullName }
  }
})
$groups = @($parts | Group-Object Index)
$present = @($groups | ForEach-Object { [int]$_.Name })
$minIndex = ($present | Measure-Object -Minimum).Minimum
$maxIndex = ($present | Measure-Object -Maximum).Maximum
$missingIndex = @($minIndex..$maxIndex | Where-Object { $present -notcontains $_ })
$missingPair = @($groups | Where-Object { @($_.Group.Stream | Sort-Object -Unique).Count -ne 2 })

$sizeAnomalies = foreach ($stream in 'fl','fh') {
  $series = @($parts | Where-Object Stream -eq $stream)
  $modeSize = [long](($series | Where-Object Index -ne $maxIndex | Group-Object Length |
    Sort-Object Count -Descending | Select-Object -First 1).Name)
  $series | Where-Object { $_.Index -ne $maxIndex -and $_.Length -ne $modeSize }
}
$sizeAnomalies | Select-Object Stream,Index,Length |
  Export-Csv (Join-Path $OutDir "size_anomalies.csv") -NoTypeInformation -Encoding UTF8

$complete = @($groups | Where-Object { @($_.Group.Stream | Sort-Object -Unique).Count -eq 2 } |
  ForEach-Object { [int]$_.Name } | Sort-Object)
$pick = @($complete[0], $complete[[math]::Floor(($complete.Count-1)/2)], $complete[-1]) |
  Sort-Object -Unique
$representatives = @($parts | Where-Object { $pick -contains $_.Index })
$hashTargets = @($files | Where-Object Name -like '*_cfg.txt') + @($representatives)
$hashTargets | ForEach-Object { Get-FileHash -LiteralPath ($_.FullName) -Algorithm SHA256 } |
  Export-Csv (Join-Path $OutDir "representative_sha256.csv") -NoTypeInformation -Encoding UTF8
$tmp = @($files | Where-Object { $_.Name -like '*.tmp' -or $_.Name -like '*.tmp.*' })

@(
  "collector=$CollectorVersion"
  "run_id=$RunId"
  "data_dir=$DataDir"
  "pair_count=$($complete.Count)"
  "index_range=$minIndex..$maxIndex"
  "missing_indexes=$($missingIndex -join ',')"
  "missing_pairs=$($missingPair.Name -join ',')"
  "size_anomaly_count=$(@($sizeAnomalies).Count)"
  "tmp_count=$($tmp.Count)"
) | Set-Content (Join-Path $OutDir "summary.txt") -Encoding UTF8

  • [PR04][STARTUP_EP4] と [PR01][CYCLE] が出る（両方無ければ build/deploy mismatch）

実験1：直結標準化＋スイッチ区間の対照

A/B の変数は「ケーブル交換」ではない

機械から出ている固定ケーブルは交換できない。真因（親プラン「タイムアウト問題の真因」節）は、その先端に挿入されていたインライン on/off 切替スイッチ区間（メス—スイッチ—オス・着脱可能）であり、オーナーが一度外して改善を確認済み。よって本実験の管理変数はスイッチ区間の着脱のみ：A＝直結（標準）／B＝スイッチ区間を再挿入（対照）。

目的判定基準（プラン確定）
「インライン切替スイッチが主因」を定量確定し、直結標準化（D16）を根拠付ける 直結で高密度×N ランが Type C（ep6Timeouts=0）、対照（スイッチ区間挿入）でのみタイムアウト再現

手順

  1. ステップ0で決めた採用候補（r7 または r18 相当）を使う。直結（スイッチ区間なし）のまま高密度サンプルで ×3 ラン。各ランの Type 判定を記録（実験4の高密度×3と共用可）
  2. SW-SEG-01（スイッチ区間）を機械ケーブル先端とホスト側の間に再挿入（スイッチは ON 位置）。ホストポート・マシン・サンプル・パラメータは不変
  3. 高密度で ×1〜2 ラン（対照）。Type A/B の再現とログ署名（DDR_WR_END/RD_END・切断有無）を記録
  4. スイッチ区間を外して直結に戻し、高密度×1で Type C 復帰を確認（可逆性）

観測ポイント

  • ep6Timeouts 値、Type A/B/C 分類、切断（USB board is removed!）の有無
  • 可能ならリンク速度（SuperSpeed か USB2 フォールバックか）。現行アプリに表示は無いためデバイスマネージャ／PIDで確認（PID 0xFFF2=SS／0xFFF3=USB2）

終了時アクション

  • 直結を標準構成として確定し、使用ホストポートと合わせて記録（D16 の運用開始）
  • スイッチ区間 SW-SEG-01 は「再現用資材」として保管（廃棄しない）

実験4：実 .bin/_cfg サンプル取得と gcsa 照合

目的判定基準（プラン確定）
gcsa 互換契約の実行時確認。D17（物理ch順＝gcsa想定順）を実サンプルで照合・凍結し、cfg 必須行・特徴量算出方針を確定する 実サンプルが gcsa v1 パーサを通り、物理ch順・Select(0/1)・A/H/W 算出方針が確定

手順（現地）

  1. 基準サンプル取得：CH1-13 全有効・既知の gain/offset・記録したトリガ設定で1データセット取得（*_fl_*.bin＋*_fh_*.bin＋_cfg.txt）。使用パラメータは判定シートに全て記録
  2. ch順特定ラン：物理ch→波形chの対応を一意化するため、1チャンネルだけ条件を変えたランを2〜3本追加取得（例：特定 FL ch のゲインのみ大きく変える／特定 ch のみ有効化）。どの ch を変えたかを必ず記録
  3. サンプル温存が難しい場合に備え、基準サンプルは最優先で取得しておく（セッション後半の機材トラブルに対するヘッジ）

手順（照合——持ち帰り後で可）

  1. _cfg.txt が gcsa 必須3フィールドを持つか：trigger_set.high_freq_num_per_sample・wave_get_set.waveformes_nums_per_file（スペルミス "Waveformes" のまま）・trigger_set.select（≥1ch が 1）
  2. .bin が契約通りか：ヘッダレス <u2・channel-major・FL 8ch／FH 5ch・2400 サンプル・exact-multiple
  3. gcsa v0.4.1 でロード → extract_all で 24 カラム（ch毎 A→H→W 順）が得られる
  4. ch順特定ランを使い、変えた物理 ch が gcsa の想定 index（FSC,SSC,FL1..FL6／fsGMI,ssGMI,flGMI,dGMI,bfGMI）に現れるかを照合 → 一致なら D17 凍結、不一致なら ch-mapping 表1枚で吸収（プラン想定の対応）
  5. ROI/BG 既定（FSC/SSC/FL4-6=1100・FL1-3=125・長250・BG[0:50]）が実波形のパルス位置と整合するか目視確認
  6. 基準サンプル一式の SHA256 を記録し、ゴールデンサンプルとして凍結・保管

ドライバ入替・ロールバック手順（詳細）

実機PCは Win11・1台のため、新ドライバ系列（実験5→条件付きで実験5b）は実験2・3・4 が完了してから実施する。PC操作に不慣れな実施者を想定し、クリック単位で記す。レジストリは触らない（実験5bが必要になった場合も配布スクリプトが自動処理する）。

入替前チェック

  • ☐ 実験2・3・4 の判定シート記入とログ回収が完了している
  • ☐ AnalogBoard アプリを終了している
  • ☐ 新旧ドライバのフォルダをローカルに用意している：新＝FX3\Win11\x64（1.3.0.4）／旧＝現行運用版（1.2.3.20 系）のフォルダ

入替（旧 → 新 1.3.0.4）

  1. 画面左下のスタートボタンを右クリックし、メニューから「デバイス マネージャー」をクリック
  2. 一覧の「ユニバーサル シリアル バス コントローラー」の左の「＞」をクリックして開き、Cypress／FX3 系の名前のデバイスを探す（見つからない場合：メニューの「表示」→「デバイス（接続別）」に切り替えて探す。機械の電源とUSB接続も確認）
  3. そのデバイスを右クリック→「プロパティ」→「ドライバー」タブ。現在のバージョン（1.2.3.20 系のはず）をメモしてから「ドライバーの更新」をクリック
  4. 「コンピューターを参照してドライバーを検索」→「参照」で用意した FX3\Win11\x64 フォルダを選択 →「次へ」
  5. 完了後、もう一度プロパティの「ドライバー」タブを開き、バージョンが 1.3.0.4 になっていることを確認して判定シートに記録
  6. デバイスが「！」マーク無しで表示されていることを確認（「！」が付いたら下の「困ったとき」へ）

ロールバック（新 → 旧）

  1. 同じ手順でプロパティの「ドライバー」タブを開く
  2. 「ドライバーを元に戻す」ボタンが押せる場合：クリックして案内に従う（理由は「以前のバージョンの方が安定していた」等でよい）
  3. ボタンが灰色で押せない場合：「ドライバーの更新」→「コンピューターを参照」→「コンピューター上の利用可能なドライバーの一覧から選択」→ 一覧から 1.2.3.20 系を選択 →「次へ」。一覧に無ければ「参照」で旧ドライバのフォルダを指定
  4. バージョンが 1.2.3.20 系に戻ったことを確認
  5. 復帰スモーク×1：AnalogBoard アプリを起動し、低密度1サイクルで Type C（ep6Timeouts=0・DDR_RD_END=1）を確認。結果を判定シート 5-4 に記録

困ったとき

  • デバイスに「！」が付いた／認識されない → USB コネクタを抜き差し。直らなければ PC を再起動
  • 更新ウィザードで「最適なドライバーが既にインストールされています」と出る → 「コンピューター上の利用可能なドライバーの一覧から選択」の経路でバージョンを明示指定する
  • どのデバイスか自信がない → デバイスを右クリック→プロパティ→「詳細」タブ→「ハードウェア ID」に VID_04B4 が含まれていれば対象

実験5b：Win11 DFX無効化 A/B（実験5失敗時のみ・スクリプトで実施）

実施条件（2026-07-06 オーナー方針：レジストリ操作は極力避ける）

実験5の first gate（接続＋アイドルEP4）が失敗した場合のみ実施する。実験5が成功していたら本実験はスキップし、判定シートに「スキップ（実験5成功）」と記録する。実施する場合もregedit は開かない——配布スクリプト（scripts\field-session\）が対象デバイスの自動検出・設定・復元まで行う。

目的判定基準（プラン確定）
Win11 新ドライバでの通信不可の主因が DFX（アイドルサスペンド）かを単変数で切り分け NoDfx（WdfDirectedPowerTransitionEnable=0）で Get ep4 register data failed. が消えるか
やってはいけないこと
  • INF（cyusb3.inf）を編集しない——署名 cat が無効化される
  • INF 同梱の CyUsb3NoDfx セクションでインストールしない——DriverEnableDfx=0 と WdfDirectedPowerTransitionEnable=0 の2変数が同時に変わり交絡する

事前準備（開発側・セッション前日まで）

  • ☐ scripts\field-session\ フォルダ一式（dfx_status.ps1／dfx_off.ps1／dfx_on.ps1／README）を実機PCにコピーしておく（例：C:\field-session\）

  2. 管理者ターミナルを開く：画面左下のスタートボタンを右クリック→「ターミナル (管理者)」（または「Windows PowerShell (管理者)」）→ 確認ダイアログで「はい」

実験6：録画キャプチャ（ログ保全＋専用診断ラン）

目的判定基準（プラン確定）
エミュレータ（Tier 1/2/3）・回帰コーパスの初期充填。特に失敗トレース（タイムアウト・切断・ショートリード）の保全 各 config・各ドライバ構成（Win11 旧/新）で EP2/4/6 の証跡を取得し、障害トレースも保全されている
フェーズ0での「録画」の実体

新通信層の Recorder はまだ存在しないため、フェーズ0の録画は①アプリログ＋②保存データ（.bin/_cfg）＋③USBレベルキャプチャの3点セットで代替する。③の標準ツールは USBPcap＋Wireshark とする。ただし USBPcap は Windows USB スタックにキャプチャ用 filter を入れるため、取得タイミングへ影響しうる。判定ランでは使わず、同条件を短く再現する専用診断ランで使う。①②は全ランで自動的に取れるので、「消さない・全部持ち帰る」が本実験の実質タスク。

手順

  1. 全実験を通じ、ラン毎に「ビルド版・OS・ケーブル・密度・パラメータ・結果 Type」を判定シートに記録（コーパスのメタデータになる）
  2. アプリログ（logs/<ver>/logs/*.log）と保存データ（*_cfg.txt・.bin ペア）を失敗ラン含め全て保全
  3. 専用キャプチャラン：判定に使わない追加ランとして、idle EP4×10、EP2 Set→EP4 read、低密度×1、失敗再現×1 を USB レベルキャプチャ付きで取得する。高密度×1は必要時のみ短時間で行い、×3／×100 soak では原則使わない
  4. キャプチャファイルはラン ID（判定シートの番号）と紐付けて保存

USBPcap＋Wireshark の使い方

  1. 事前準備：実機セッション前に Wireshark を USBPcap オプション付きでインストールし、必要なら再起動する。インストールや再起動は実機時間中に行わない
  2. 保存先作成：例 C:\AnalogBoardField\usbpcap\YYYYMMDD\ を作り、判定シートと同じラン ID で管理する
  3. 対象 bus の確認：Wireshark を起動し、USBPcap の interface 一覧から Cypress/FX3 デバイス（VID 04B4 系）がぶら下がる root hub を選ぶ。特定が難しい場合は準備時だけ抜き差しして増減する bus を確認し、判定ラン中には抜き差ししない
  4. 判定ランを先に実施：first gate、ラダー、soak の Pass/Fail 判定は USBPcap を停止した状態で行い、ログと保存データを先に確保する
  5. 診断 capture 開始：Wireshark で該当 USBPcapX interface の capture を開始し、すぐに 1 操作だけ実施する（例：idle EP4×10、EP2 Set→EP4 read、低密度 1 cycle、失敗再現 1 回）
  6. すぐ停止して保存：操作が終わったら capture を停止し、usbpcap_E5_firstgate_idle_ep4x10_pass_YYYYMMDD_HHMM.pcapng のように実験番号・操作・結果・日時が分かる名前で保存する
  7. メモを残す：判定シートに pcapng のパス、Wireshark/USBPcap の版、選んだ USBPcapX、対象操作、開始/終了時刻、EP2=0x02・EP4=0x84・EP6=0x86 の観測有無を書く
  8. 確認：Wireshark ではまず usb 表示フィルタで全体を見て、必要に応じて endpoint address や device address で絞る。Wireshark の USB field 名は版差があるため、現地では画面上の列名に合わせてフィルタを組む
USB capture の扱い

USBPcap は診断用の補助証跡であり、Pass/Fail の主証拠ではない。Pass/Fail はキャプチャなしの判定ランで決める。Microsoft USB ETW はドライバ stack・enumeration・port state の補助確認には使えるが、EP2/EP4/EP6 payload の代替にはしない。ハードウェア USB アナライザがある場合は非侵襲のため優先してよいが、機種・設定・接続位置を必ず記録する。

コーパスに揃えるべき内訳（プラン「録画コーパスに含めるもの」より）

  • EP6 生バイト列（=保存 .bin ペア＋キャプチャ）
  • EP4 レジスタ推移（=ログの STARTUP_EP4/EP4読取行＋キャプチャ）
  • EP2 コマンド列（=Set 時のログ＋キャプチャ）

  • config タグ（=_cfg.txt＋判定シートのメタデータ）
  • 障害イベントの実トレース（=Type A/B ランのログ一式）

注意事項

落とし穴（負の知識）

High計装・観測がタイミングを変える
DebugView／デバッガ接続は OutputDebugStringA のコストを跳ね上げ、取得タイミングを変えて別の障害を「作り出す」。判定ランでは必ず外す。USBレベルキャプチャも判定ランでは使わず専用ランに分離する。
Highdeploy mismatch のまま実験を進める
マーカー（[PR04][TIMEOUT_RECOVERY] 等）が出ない状態で得た結果は解釈不能になり全ランが無駄になる。ステップ0を必ず通し、疑わしければ即座にビルド配置へ戻る。
MediumA/B の交絡
スイッチ区間の着脱時にホストポートも変える、DFX を INF の NoDfx セクションで入れる（2値同時変更）、実験間でサンプル密度が変わる——いずれも切り分け力を失う。1実験1変数を厳守し、変えたものを判定シートに書く。
MediumRD_WAIT 固着
FPGA がドレイン中に RD_WAIT で固着した場合、ホスト側だけでは回復不能（FW に abort レジスタなし）。発生したら電源サイクルで復帰し、発生時のログを障害トレースとして必ず保全する（自動再接続の上限を示す一次証拠になる）。
