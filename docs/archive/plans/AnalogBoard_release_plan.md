# AnalogBoard 製品版リリース計画

## 目的
- Test版実装を製品版品質まで引き上げる。
- 最低条件として「クラッシュしない」「データ破損しない」「再接続で復帰できる」を満たす。

## 前提
- 現状は Test版相当であり、製品版リリース可否は `No-Go`。
- まず P0 安定化を完了し、次に P1 堅牢化、最後に判定試験を実施する。

## Phase 0: 仕様凍結と受け入れ基準合意（2-3日）
- 出力仕様、USB再接続仕様、エラーハンドリング仕様を `v2` として凍結。
- Go/No-Go 基準を確定。
- 成果物:
  - `Release Spec v2`
  - `Test Exit Criteria`

## Phase 1: P0安定化実装（1.5-2週間）
1. UIスレッド以外からのUI操作を禁止し、`PostMessage` 等へ統一。
2. 終了時のスレッド停止・待機（join相当）を実装。
3. `EP6_GetData` を `buffer + size` 受けに変更し、サイズ検証を必須化。
4. DLL API から `_msize` 依存を排除。
5. `OneWaveSize==0`（CH未選択/範囲0）を事前バリデーションで遮断。
6. USB抜き差し時の connect/disconnect と転送処理の排他制御を実装。
7. `CreateEvent` などのハンドルリークを解消（RAII推奨）。
- 成果物:
  - P0修正パッチ
  - 回帰テスト結果

## Phase 2: P1堅牢化実装（1-1.5週間）
1. CSV取込の列数/型/範囲チェック追加（壊れたCSVで落ちない）。
2. 設定ファイルにバージョンを追加し、互換ルールを定義。
3. エラーコード体系を整理し、ログを運用可能な粒度へ統一。
- 成果物:
  - 設定互換仕様
  - 異常系テスト結果

## Phase 3: 検証・リリース判定（1週間）
1. 24-72時間連続運転試験。
2. USBホットプラグ反復試験（目標: 500回）。
3. 長時間取得でデータ破損/欠落/クラッシュがないことを確認。
4. 異常系（保存先不足、抜線、不正CSV）で安全停止を確認。
- 成果物:
  - テストレポート
  - Go/No-Go 判定記録

## 担当分割（最小体制）
1. App担当
- UIスレッド化、終了シーケンス、CSV堅牢化
2. DLL担当
- API安全化、USB排他、メモリ/ハンドル管理
3. QA担当
- 長時間試験、異常系試験、自動/半自動回帰
4. PM/PL
- 仕様凍結、判定会運営、リスク管理

## Go判定条件（必須）
1. 連続運転試験でクラッシュ/ハング 0件。
2. USB再接続復帰率 100%。
3. 取得データ破損 0件。
4. 不正入力時に安全失敗（プロセスダウンしない）。

## 初週の実行順
1. P0チケットを6件に分割して起票。
2. UIスレッド化 + 終了待機を先行実装。
3. DLL API安全化（サイズ引数必須化）を実装。
4. USB排他 + ハンドルリーク修正を実装。
5. スモーク回帰を実施し、残課題を棚卸し。

## 目安スケジュール
- Week 1: Phase 0 完了 + Phase 1 着手
- Week 2: Phase 1 完了
- Week 3: Phase 2 完了
- Week 4: Phase 3 判定試験 + リリース可否決定

## リスクと対策
1. USB再接続のタイミング競合
- 対策: 状態遷移を明示し、単一スレッド/排他管理で直列化。
2. 既存CSVとの互換性崩れ
- 対策: バージョン判定 + 後方互換ローダーを用意。
3. 長時間試験の工数不足
- 対策: 夜間バッチ化・自動収集ログで再現性確保。

## リリース判断
- Phase 1 未完了の時点では `No-Go`。
- Go判定条件を全て満たした場合のみ製品版へ進行する。
