## 詳細フィードバック

### 重要度: High（スコア 0-3）

#### API設計の完全性（2/10）

**問題:** `WaveAcquisitionEngine` 導入やキュー導入の方針はあるが、実装者が迷わないレベルのインターフェース契約がない。特に Start/Stop の戻り値、エラー分類、タイムアウト時の復帰条件が未定義。  
**改善案:** 「インターフェース設計」セクションを追加し、公開メソッド・戻り値・エラーコードを固定する。

```markdown
## インターフェース設計

### WaveAcquisitionEngine
- `INT Start(const AcquisitionConfig& config);`
  - Returns: `USB_SUCCESS` / `USB_ERR_*`
- `INT Stop(DWORD timeoutMs = 5000);`
  - timeout時は `USB_ERR_THREAD_STOP_TIMEOUT`
- `EngineStatus GetStatus() const;`

### BlockingQueue<WaveChunk>
- `bool Enqueue(WaveChunk&& chunk, DWORD timeoutMs);`
- `bool Dequeue(WaveChunk& out, DWORD timeoutMs);`
- Full時: `timeoutMs` 超過で false を返却
```

#### セキュリティ（2/10）

**問題:** 本件はネットワークAPIではないが、ファイルパス生成・ログ出力・異常時ダンプで機密/整合性を損なうリスクがある。対策方針が未記載。  
**改善案:** 「セキュリティ」セクションを追加し、少なくともファイルパス検証、ログ制御、異常ファイル保護方針を定義する。

```markdown
## セキュリティ
- 出力先パスは許可ディレクトリ配下のみ許可（`..` を拒否）。
- ファイル名に制御文字/予約名を含む場合は保存を拒否。
- デバッグログには生データを出力しない（サイズ・ハッシュのみ）。
- `.tmp` は `Publish` 成功時のみ確定、失敗時は監査用に隔離保存。
```

#### マイグレーション・後方互換性（3/10）

**問題:** 既存処理を段階的に置き換える際の切り戻し条件と手順が不明。Phase 2 以降で障害が出た場合の運用手当が不足。  
**改善案:** 段階リリースとロールバック条件を追記する。

```markdown
## 移行・ロールバック計画
- Step 1: Phase 1 を feature flag なしで適用（DLLバグ修正のみ）。
- Step 2: Phase 2 は `UseAsyncWriter` フラグで切替可能にする。
- Rollback条件:
  - timeout率が baseline +0.05%以上悪化
  - ファイル欠損 1件以上
- Rollback手順:
  1. `UseAsyncWriter=0` に戻す
  2. 直近ビルドへ差し戻し
  3. `.tmp` を隔離し再処理キューへ移送
```

### 重要度: Medium（スコア 4-6）

#### 必須セクションの網羅性（6/10）

**問題:** 必須観点が複数フェーズに分散しており、レビュー/運用で参照しにくい。  
**改善案:** `セキュリティ` `依存関係` `移行・後方互換性` を独立見出しで追加する。

#### スコープの明確性（6/10）

**問題:** 「今回やること」は明確だが、今回やらない内容（UI刷新、Rust本採用条件、既存通信仕様変更の有無）が明記されていない。  
**改善案:** `Out of Scope` を追加し、今回除外項目を列挙する。

#### テスト方針・テストケース（6/10）

**問題:** テスト項目が方針レベルで、再現条件と合否判定値が不足。  
**改善案:** ケースごとに「入力条件/注入条件/期待結果/KPI」を表形式で定義する。

#### 技術選定の妥当性（6/10）

**問題:** Rust案の比較軸（性能、保守性、導入コスト）が未定義で意思決定できない。  
**改善案:** C++継続案 vs Rust導入案の比較表（採用条件・見送り条件）を追加する。

#### 依存関係の明示（4/10）

**問題:** 追加予定の依存（標準C++機能、将来Rust crate、テスト用モック方式）の記述が粗い。  
**改善案:** 依存一覧に「対象モジュール/バージョン方針/ライセンス確認有無/導入理由」を追加する。

### 重要度: Low（スコア 7-8）

- 内部アーキテクチャ、実装計画、既存コード整合性は良好。  
- ただし KPI 判定で「測定期間」「測定時の負荷条件」「失敗率の信頼区間」を追加すると再現性が向上する。
