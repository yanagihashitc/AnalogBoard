エグゼクティブサマリ

オーナーの問い——「将来的に AnalogBoard と sys_app をひとつのアプリにまとめる案はどうか。統合するなら今の設計を変えておく必要があるか」——に対し、 再構築プラン（Draft 1.12）・下流改変プラン（Draft 0.4）・sys_app（約29万行）・gcsa の実装を精査し、 製品/デプロイ・リアルタイム信頼性・契約ミニマリストの3つの独立した設計視点で統合案を作成、 固定決定 D1〜D18 とコード実態への敵対的検証を通して統合した。

結論：コードベースの統合（単一プロセス・単一コードベース化）は非推奨。3視点すべてが独立に同じ結論に達した。 ただし「ユーザーにとってのひとつのアプリ」はコード統合なしで達成でき、その到達形は「一製品・複数プロセス」 ——同梱配布＋取得完了からの自動ハンドオフ（近期）、必要なら WPF シェルへの sys_app web UI 埋込（将来のアドオン）。 統合レイヤの実体はすでに設計済みの gcsa ネイティブ Zarr ストア＋正準 gate v1＋meta.json 契約そのものであり、 これを完成させることが最良の「統合保険」になる。

今から入れるべき設計変更は6件（下記）。うち2件は時間制約付き： ②finalization／パーティション可視性契約は meta.json（唯一「安定度：中」の契約項目）の凍結前に、 ③パーティションシャーディング方式はフェーズ0の D12 往復検証と同時に決める必要がある。 ②は統合と無関係に、今日の正しさの穴（取得中 import によるイベント数のサイレント切詰め）も塞ぐ。

キーコンセプト

一製品・複数プロセス
ユーザーには1つの製品（同梱インストール・一貫した動線）に見せつつ、取得（WPF＋ネイティブコア）と解析（sys_app スタック）は別プロセス群に保つ形。取得の障害ドメインを解析から隔離したまま「ひとつのアプリ」体験を実現する。
ストア契約＝統合レイヤ
gcsa ネイティブ Zarr ストア＋meta.json＋正準 gate v1。両アプリが従う唯一のデータ語彙であり、将来アプリをどう包み直しても変わらない座標系。契約を完成させること自体が最大の統合保険。
射影（projection / S1〜S8）
sys_app が import 時にストアを自前の Parquet/JSON 面へ複製する方式（下流改変プラン。実行保留中）。sys_app の可視化・gating 層を無改修に保つために選ばれた——統合しても「seam の位置」が変わるだけで再計算禁止（D5）等の不変条件は残る。
finalization マーカ
データセットが「取得完了」かを外部リーダが判定できる meta.json 上の明示フラグ＋パーティションマニフェスト。gcsa の現行慣習（meta.json 最終書込＝暗黙のコミット）は再アーム append と両立しないため、明示マーカが正しい代替となる。
C ABI 中立性
ネイティブコアと UI の境界を「版付き POD 構造体のみ・フレームワーク型なし」に保つ規則。同一コアを WPF（現行）・常駐サービス・将来の web フロントのいずれからもホスト可能にする。
WebView2 埋込
WPF ウィンドウ内に Edge ベースのブラウザペインを置き、localhost の sys_app web UI を表示する方式。sys_app 側の改修が原則不要な「見た目の統合」の最安経路。事前準備なしに後付け可能（＝今は作らない）。

結論：統合の是非と到達形

統合オプションの比較

案内容判定根拠
1. 二アプリ＋自動ハンドオフ 取得完了（atomic finalize）→ sys_app へ自動登録（既存 POST /api/v1/datasets/import_from_path）→ 通知センターから「sys_app で開く」 近期の推奨形 sys_app 改修は自動登録トリガのみ（UX-16 のオーナー確認後に S プロンプト化）。フェーズ4まではこの形で十分
2. WPF シェル＋WebView2 埋込 WPF が localhost の sys_app web UI をホストし「一画面・一製品」に見せる。案1の上に純追加 将来のアドオン Next.js は same-origin の /api rewrite proxy を持ち CORS 不要で埋込可。sys_app のコード変更が原則不要。事前準備なしに後付けできるため今は作らない
3. 取得コアをサービス化し web UI へ一本化 取得制御・ライブ表示を含む全 UI を sys_app 側 web スタックに移す 却下 D1（WPF 確定）と衝突。sys_app は push 経路（WebSocket/SSE）ゼロのバッチ型でライブ表示を担えない。FR-11（機械スイッチ外部開始）が常駐ネイティブを要求
4. 単一コードベース統合 sys_app 機能の WPF 書き直し、または取得の web 化 却下 ~29万行（TS/TSX 945ファイル＋Python 167ファイル）＋解析数値核は gcsa（Python+Rust）に在る。再実装コストに見合う便益なし

なぜコード統合が成立しないか（検証済みの3つの事実）

  • 実行環境の非対称：sys_app は Docker/Linux 専用（Windows 実行は WSL2＋Docker Desktop を README が必須指定）。ネイティブ Windows 実行経路は dev/production いずれの構成にも存在しない。
  • 取得側の常駐・隔離要件：計測は機械側スイッチから開始される（FR-11）ため取得コアは常駐ネイティブ必須。かつ 200MB/s ホットパスは過去3回「テスト緑のまま」現場退行しており（CRT アロケータ変更×2・エンジン抽出・ホットパス計装）、解析ワークロードと同一障害ドメインに置くことは信頼性上許されない。
  • 解析核の言語境界：バイナリ解析・特徴量・gate マスク・補償・UMAP/SVM は gcsa（Python＋Rust 拡張）に実装されており、C#/WPF への移植は数値核の再実装か新たな interop 境界を意味する——いま切ってある C ABI と同種の seam を別の場所にもう1本引くだけで、統合の便益がない。
プランは既に「統合保険」を多く含む

単一データ基軸(gcsa ストア)・C ABI 隔離（コア別プロセス化の選択肢を明記済み）・D5/D14/G2（記録一回・再計算禁止・版管理）・正準 gate v1＋生ADC線形座標・UX-17（安定 dataset id）・UX-20/FR-13（sys_app の可視化コンポーネントを明示的な先行例として参照）——機能面の収斂は設計段階から起きており、将来の製品統合の概念コストは既に低い。

gcsa（共有解析ライブラリ・ストアの権威実装）

項目事実統合への含意
性格GUI なしの importable Python ライブラリ＋任意 Rust(PyO3) 拡張（A/H/W・polygon gating・binary parse 等の高速路）統合形態に依らず「共有数値核」として維持するのが正解。Rust バイナリは Python 版毎ビルドの配布注意
完了概念finalized フラグ・パーティションマニフェスト不在。streaming writer は全 append 後に meta.json を書く＝「meta.json 最終書込」が暗黙のコミットポイントで、書きかけ dataset はリーダから不可視AnalogBoard は再アーム append のため meta.json を先に書く＝この暗黙規約が壊れる。明示 finalization マーカが正しい代替（→変更②）
並行安全性docstring は「Cross-process access」を謳うが実体はプロセス内 RLock のみ。ファイルロックなし。append 経路は atomic-rename の保護外「取得中に別プロセスが読む」は現状未サポート。単一書き手規則＋可視性マーカで契約化する（→変更②）
リーダ検証validate_partition_set が 0..n-1 の完全集合と正確な round-robin 分配式を要求。全読みは途中状態で fail-loud、スライス読みは長さ検証をバイパス（サイレント不整合の窓）パーティションレイアウト変更＝リポジトリ横断の契約変更。シャーディング方式はフェーズ0で決着必須（→変更③）

sys_app との関係sys_app は現在 gcsa ZarrStore を読んでいない（cfg/bin を lazy-import アダプタ経由で再取込し、独自の ad-hoc zarr を書く）。S1〜S8 が繋ぐ計画で実行保留中射影 ETL（S1/S3/S4/S5）は「二アプリ前提」の投資。統合方針を早く決めるほど捨てコードを避けられる——ただし G1〜G3 と契約系決定（D13/D14/D15）はどの形態でも価値が残る

① C ABI 中立性ルールの明文化

C ABI を越えるのは版付き POD／シリアライズ可能構造体のみとする規則をプランのアーキテクチャ節に明記する。 コマンド（start/stop・一括設定適用・レシピ）とフィードフレーム（ビニング済み scatter・histogram・カウンタ・GMI スナップショット ≤100×2400・リンクヘルス）を列挙し、 .NET/WPF/COM 型とコールバックのオブジェクトグラフを禁止、コールバックは frame-ready 通知＋pull か polled queue に限定する。 プランは既に「集計のみ越境」「必要ならコア別プロセス化」と示唆しているが、規則としては書かれていない。 これ1つで同一コアが WPF（現行）・常駐サービス・将来のフロント差替えのすべてに対応する——最安の統合保険。D1 と無衝突。

② finalization／パーティション可視性契約 凍結前必須

3視点が独立に導出した最重要項目。出力契約の凍結前に meta.json へ additive に入れる：

  • status: open|finalized・finalized_at・単調増加の write_generation
  • パーティションマニフェスト：配列毎（pulse_features / gmi_waveform / fl_waveform）の現在行数と per-partition sealed フラグ
  • 単一書き手規則：datasets/<id>/ 内の書き手は取得コアのみ（sys_app 派生面は {source_dir}/sys_app/store/ に既に分離されていることを契約に昇格）
  • finalize イベント：atomic finalize 時に writer スレッドで発火（dataset id＋ストアパス）。transport 中立——ネイティブコアから sys_app への HTTP 直結は検証で却下済み
  • ストアフォルダ＋meta.json を唯一のクロスアプリレジストリと明記。派生面は (dataset_id, feature_schema_version, write_generation) をキーとするキャッシュと位置づける

根拠：(a) 現行計画のままでは 今日の正しさの穴がある——S5 は event_count を pulse_features.shape[0] から導出するため、 取得中（再アーム append 中）に import するとエラーなしで切詰められたイベント数が確定する。temp→atomic rename は「1ファイルのバイト列」しか守らず、複数パーティション＋複数配列＋metaの横断整合は守らない。 (b) gcsa の暗黙 finalization（meta.json 最終書込）は、再アーム append のため meta.json を先に書く AnalogBoard の書込順と両立しない——明示マーカが正しい代替。 (c) G1/S1/S5 プロンプトからこのマーカを参照するよう下流改変プランの固定入力契約表にも反映し、フェーズ0往復で gcsa が additive フィールドを許容することを確認する。

③ パーティションシャーディング方式の決着（フェーズ0）

プランのストア仕様は round-robin（e % n_partitions）を踏襲するが、この方式は全パーティションが同時に成長するため 「ここまでは確定」という sealed prefix が存在せず、取得中の追従読取・安全な途中 import に構造的に不利。 さらに gcsa のリーダ検証は正確な round-robin 分配式そのものを検証しており、レイアウト変更は後からだとリポジトリ横断の契約変更になる。 選択肢は (a) append-sequential＋seal（partition p を埋めて封印し p+1 へ）へ変更、(b) round-robin 維持＋マニフェストの per-partition sealed で緩和、の2つ。 どちらを選んでも既計画の phase-0 D12 往復検証（gcsa が実ロードできること）を通すことが非交渉条件。紙の上の今が最後の安い変更点。

④ フィード fan-out ポイントの設計時定義

集計フレームは BlockingQueue の消費側（EP6 読取・ZarrWriter スレッドの外）で生成し、 sequence number＋monotonic timestamp を付与、multi-sink の attach point を設計上1箇所命名する（現在の sink は C ABI の1本のみ。第2 sink は作らない）。 将来 web フロントや監視ミラーを足すときにホットパスへ触れない構造を、コードでなく設計で保証する。 統合と無関係の即時価値：シーケンス番号により Tier1/2 回帰でフィード再生が決定的になる。

⑤ Tier1 非WPF ホスト適合テスト

最小の非WPF ホスト（素の C/C++ またはコンソール）から C ABI 経由で start→取得（エミュレート）→再アーム→stop→ストア書込の全周を、 シリアライズされたコマンド／フレームだけで駆動するテストをフェーズ1テスト計画に追加。 ①の規則を「レビューで守る」から「コンパイル／テストで守る」に変える。既計画の Tier1 インプロセスハーネスとコストの大半を共有。

⑥ デプロイトポロジ節＋共存 failure-domain NFR＋オーナー質問

プランに短い節を追加：(A) 計測PC同居（WPF＋WSL2/Docker sys_app）vs (B) 分離（計測PC＋解析ワークステーション）を列挙し、 WSL2/Docker の install・update・support の所有者を命名、出荷既定の選択をフェーズ4受入の明示入力にする （同居 soak：200MB/s 持続取得＋sys_app import 並走）。あわせて NFR に共存規則（実装ではなくルールとして）： 解析プロセスは取得より低い CPU/IO 優先度・同居時は .wslconfig で WSL2 メモリ上限・取得中の重ジョブ抑止・ ディスク残監視（UX-11）を派生面の複製footprint＋Zarr キャッシュ（既定50GB）込みへ拡張。 P1 注記（同居構成では平文射影面が保護ストアの隣に置かれる）もここに記録する。 この節が下記オーナー判断事項の受け皿になる。

追補：sys_app の Tauri デスクトップ化（実現性評価）

オーナー方針（2026-07-09）

本編の結論を受け、オーナーより「sys_app は ML アルゴリズム追加等で頻繁に更新されるため別アプリ維持が望ましい。また sys_app の Tauri によるデスクトップアプリ化を視野に入れている」との方向性が示された。本節はその実現性を sys_app / gcsa の実コードに対する追加調査（3エージェント・file:line 検証付き）で評価したもの。

評価：方向性は妥当。Tauri 化は実現可能——ただし工数の実体は「Python バックエンドのサイドカー化」が8割

別アプリ維持は本編の結論（3視点独立一致）と完全に整合する。リリースサイクルの非対称——取得側は実機ゲート付きの安定重視、解析側は頻繁更新——はまさに「versioned なストア契約で疎結合に保つ」ことで吸収する設計であり、 feature_schema_version（D14）・store_format・gate 正準v1 がその仕組みそのもの。DR/ML タスクの計算本体が現状 TODO スタブである事実も「これから頻繁に足していく」という前提を裏づける。

Tauri 化については、Tauri（Rust シェル＋WebView2）が担えるのはフロントエンドの器と updater・deep-link 程度で、 本体は Python バックエンド（FastAPI＋gcsa）をネイティブ Windows のサイドカー実行ファイルに固めること。 この難所を3領域に分けて実コードで検証した結果、いずれも「困難」ではなく「列挙可能な作業」だった。

領域検証済みの現状Tauri 化に必要な作業
① ジョブ基盤
（Celery/Redis の除去）		 結合は驚くほど薄い：タスクは全6個（うち UMAP/t-SNE/SVM/LightGBM/predict の4個は TODO スタブ、実装済は dataset.export のみ）。apply_async 呼出は全コードで2箇所（job_submission_service.py:102・dataset_export_service.py:79）。ジョブ状態・進捗・結果はすべて SQLite（Celery result backend は一度も読まれない）。retry/ETA/chord/revoke 等の高度機能は不使用。Redis は実質ブローカ専用（メタデータキャッシュは既定 in-memory）。フロントは 2秒 HTTP ポーリングで transport 変更不要 in-process ジョブランナー（ThreadPool/subprocess、ml_light/ml_heavy/dr_heavy レーンと time limit を再現）を書き、enqueue 2関数＋Celery 依存のヘルスエンドポイントを差替え。注意：Celery 不在時の既存フォールバック（TaskStub）は enqueue を黙って握り潰すテスト専用実装であり、そのままでは使えない

決めが必要な2点

  • sklearnex / Intel oneDAL：現行 Docker イメージは oneDAL ネイティブライブラリなしでは gcsa の import に失敗する構成。デスクトップ版で (a) 大容量の oneDAL DLL 群を同梱するか、(b) sklearnex を除外するか——除外すると sklearnex 直列化済み SVM アーティファクトの読込が ModelArtifactError になるため、既存モデル資産の扱いとセットで判断が要る。
  • /datasets/[id] ルートの再構成方式：query-param 化（既存 /workspace パターン）か、プレースホルダ param＋クライアント側解決か。

AnalogBoard 側との相乗効果

  • 計測PCから WSL2＋Docker Desktop が消える——本編⑥（デプロイトポロジ）の同居案が抱えていた最大の負担（vmmem メモリ競合・9p I/O ペナルティ・Docker Desktop ライセンス・サポート所有者問題）がまとめて解消する。
  • DATA_DIR のコンテナパス変換問題が消滅——ハンドオフは Windows ネイティブパスの import_from_path で直に成立。finalize イベント→自動登録の実装が単純化する。
  • Tauri の組込み updater は「頻繁に更新する解析アプリ」という運用要件そのものに合う（Windows コード署名の準備は必要）。deep-link プラグインは通知センターからの「sys_app で開く」動線に使える。
  • ただし共存 failure-domain 規則（本編⑥）は引き続き必要——Docker が消えても、取得中の CPU/IO 競合（特に import・ML ジョブ）と、finalization マーカ（本編②）の必要性は変わらない。むしろ同居が容易になるぶん「取得中に import される」確率は上がる。

進め方の推奨

Tauri 化は AnalogBoard のフェーズ0〜4 とも S1〜S8 とも独立しており、今決める必要はない。ただし安い先回りが2つ： (1) 下流改変プラン実行時（S プロンプト）に Celery seam を今より深くしない・Redis 依存を増やさない方針を一文添える（現状の薄さが Tauri 化の最大の資産）。 (2) DR/ML の計算本体（現状スタブ）を実装する際、ランナーを Celery 非依存の関数として書き enqueue 層と分離しておく（既にその形の seam：UMAP_RUNNER 等のモジュールレベル差替え点がある）。

調査で見つかった既存の問題（Tauri と無関係・sys_app 側への展開推奨）
  • export ジョブのキュー配線ギャップ（要確認）：dataset.export は task_routes のどのパターンにも一致せず既定 celery キューへ入るが、prod/dev の worker は -Q dr_heavy,ml_light,ml_heavy しか購読していない——Docker 構成では export ジョブが永久に queued のままになる可能性（構成からの推定・未実行検証）。
  • cancel の実効性：DELETE /jobs/{id} は SQLite 上で cancelled にするだけで、実行中タスクは中断されない。さらに完了時 set_result() が無条件に completed へ上書きし、実行中キャンセルを黙って取り消す。
  • import の同期実行：最重量ワークロードの dataset import が async エンドポイント内で同期実行され、完了までイベントループを塞ぐ。デスクトップ化（単一プロセス）ではジョブランナーへ移すべき。
