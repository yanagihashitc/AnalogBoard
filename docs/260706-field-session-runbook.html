概要

本書は親プラン（docs/plans/260703-analogboard-rebuild-plan.html「まとめ実機セッション」節）で定義されたフェーズ0のまとめ実機セッションを、現地で迷わず流し切るための実施手順書である。実機アクセスは希少なため、事前にビルド済みバイナリ・本手順書・判定シートを全て用意し、1セッションで完了させる。セッションの成果（判定結果・録画コーパス・実サンプル）は、D4（移植ベースライン採用）・D17（物理ch順凍結）・ドライバ戦略（Plan A/B）・4GB戦略の判断入力になる。

前提と原則

全ラン共通条件

  • ビルドは Release | x64。Visual Studio デバッガ非接続、DebugView を起動しない（DLL 側 OutputDebugStringA が高コスト化し取得タイミングを変える——過去の計装退行の教訓）
  • ホットパスログを無効化した計測用改造ビルドは使わない（本セッションは配布した標準ビルドのみ）
  • A/B 比較は単変数で行う：同一マシン・同一USBポート・同一サンプル・同一取得パラメータのまま、比較対象（ケーブル／レジストリ値／バイナリ）だけを変える
  • ラン毎の証跡はビルド版別フォルダに保存：ログ＝logs/<ver>/logs/*.log、データ＝data/<ver>/*_cfg.txt・*_fl_*.bin／*_fh_*.bin
  • 失敗ランのログ・データも絶対に消さない（障害トレースは録画コーパスの最重要資産）
  • ラン開始・終了ごとに判定シートへ即時記録（記憶に頼らない）

判定語彙

Type A
取得中の EP6 timeout。サイクル終了時も DDR_WR_END=0, DDR_RD_END=0。Wait fpga write completed. Timeout や切断を伴うことがある。
Type B
ドレイン後の EP6 timeout。DDR_WR_END=1, DDR_RD_END=0。Wait acquisition completed. Timeout・EP4 fail・切断を伴う。
Type C
正常完了。ep6Timeouts=0・DDR_WR_END=1, DDR_RD_END=1。本セッションの多くの判定基準はこれ。
良接続構成（直結）
機械から出ている固定ケーブル（交換不可）から、先端に挿入されていたインライン on/off 切替スイッチ区間（メス—スイッチ—オス・着脱可能）を外した直結状態（D16）。本セッションで標準化する。外したスイッチ区間は SW-SEG-01 とラベルして対照実験用に保管。
DFX / Directed Power
Win11 新ドライバ INF が有効化する WDF のアイドルサスペンド機構。新ドライバ通信不可の最有力仮説。レジストリ単変数 A/B で検証する（実験2）。
deploy mismatch
現地バイナリに想定の fix が入っていない状態。期待するログマーカーが出ない時は結果を解釈する前にこれを疑う（ステップ0参照）。

実験順の理由

実験1（スイッチ区間）を最初に置くのは、直結（スイッチ区間なし）で高密度が安定することを確定させないと、以降の全実験の失敗が物理層起因かソフト起因か切り分けられなくなるため。実験4のラダーは段階制（×1で確認→×3→×100）で、途中失敗時は先に進まず切り分けに切り替える（実験4の分岐参照）。新ドライバ系列は実験3（レジストリ不要）を先行させ、実験2（レジストリ変更）は実験3の first gate 失敗時のみの条件付きとする——レジストリ操作を極力避けるオーナー方針（2026-07-06）。

ステップ0：スモーク＆deploy検証

目的：配布バイナリが「想定の fix が全部入った版」であることをログマーカーで確認し、以降の実験結果を解釈可能にする。ここを飛ばすと、当日の全データが「どのビルドの挙動か不明」になり無効化しうる。

手順

  1. 主測定PC（Win11・既存ドライバ構成のまま）に dev r18相当ビルドを配置。スイッチ区間を外した直結で、決めたホストポートに接続
  2. 低密度サンプルで 1〜3 サイクル取得（Manual Get ON・既定パラメータ）
  3. ログを開き、下記マーカーを全て確認

確認マーカー（deploy検証）

  • [PR04][ENGINE_CONTEXT]・[PR04][ENGINE_ENTER]・[PR04][ENGINE_EXIT] が出る
  • [PR04][STARTUP_EP4] と [PR01][CYCLE] が出る（両方無ければ build/deploy mismatch）
  • One time max size=31211520byte（124813312byte のままなら 1-file burst cap fix 未反映）
  • saved wave count 0／immediate exit が出ない
  • DDR_RD_END=1・ep6Timeouts=0 で正常完了（Type C）
  • 完了ペアのみ *_fl_*.bin／*_fh_*.bin として残り、.tmp 残置がない
  • 失敗させた場合は _cfg.txt 末尾に Run Status / Error Code / Published Pairs / Saved Wave Count が追記される（cfg append fix）
NG時

マーカー欠落＝deploy mismatch。実験に進まず、配布物のビルド日時・ハッシュを確認して再配置する。ENGINE_EXIT error=-21001 が出る場合は parent binding fix 未反映を疑う。

実験1：直結標準化＋スイッチ区間の対照

A/B の変数は「ケーブル交換」ではない

機械から出ている固定ケーブルは交換できない。真因（親プラン「タイムアウト問題の真因」節）は、その先端に挿入されていたインライン on/off 切替スイッチ区間（メス—スイッチ—オス・着脱可能）であり、オーナーが一度外して改善を確認済み。よって本実験の管理変数はスイッチ区間の着脱のみ：A＝直結（標準）／B＝スイッチ区間を再挿入（対照）。

目的判定基準（プラン確定）
「インライン切替スイッチが主因」を定量確定し、直結標準化（D16）を根拠付ける 直結で高密度×N ランが Type C（ep6Timeouts=0）、対照（スイッチ区間挿入）でのみタイムアウト再現

手順

  1. 直結（スイッチ区間なし）のまま高密度サンプルで ×3 ラン。各ランの Type 判定を記録（実験4の高密度×3と共用可）
  2. SW-SEG-01（スイッチ区間）を機械ケーブル先端とホスト側の間に再挿入（スイッチは ON 位置）。ホストポート・マシン・サンプル・パラメータは不変
  3. 高密度で ×1〜2 ラン（対照）。Type A/B の再現とログ署名（DDR_WR_END/RD_END・切断有無）を記録
  4. スイッチ区間を外して直結に戻し、高密度×1で Type C 復帰を確認（可逆性）

観測ポイント

  • ep6Timeouts 値、Type A/B/C 分類、切断（USB board is removed!）の有無
  • 可能ならリンク速度（SuperSpeed か USB2 フォールバックか）。現行アプリに表示は無いためデバイスマネージャ／PIDで確認（PID 0xFFF2=SS／0xFFF3=USB2）

終了時アクション

  • 直結を標準構成として確定し、使用ホストポートと合わせて記録（D16 の運用開始）
  • スイッチ区間 SW-SEG-01 は「再現用資材」として保管（廃棄しない）

ドライバ入替・ロールバック手順（詳細）

実機PCは Win11・1台のため、新ドライバ系列（実験3→条件付きで実験2）は実験4・5・7 が完了してから実施する。PC操作に不慣れな実施者を想定し、クリック単位で記す。レジストリは触らない（実験2が必要になった場合も配布スクリプトが自動処理する）。

入替前チェック

  • ☐ 実験4・5・7 の判定シート記入とログ回収が完了している
  • ☐ AnalogBoard アプリを終了している
  • ☐ 新旧ドライバのフォルダをローカルに用意している：新＝FX3\Win11\x64（1.3.0.4）／旧＝現行運用版（1.2.3.20 系）のフォルダ

入替（旧 → 新 1.3.0.4）

  1. 画面左下のスタートボタンを右クリックし、メニューから「デバイス マネージャー」をクリック
  2. 一覧の「ユニバーサル シリアル バス コントローラー」の左の「＞」をクリックして開き、Cypress／FX3 系の名前のデバイスを探す（見つからない場合：メニューの「表示」→「デバイス（接続別）」に切り替えて探す。機械の電源とUSB接続も確認）
  3. そのデバイスを右クリック→「プロパティ」→「ドライバー」タブ。現在のバージョン（1.2.3.20 系のはず）をメモしてから「ドライバーの更新」をクリック
  4. 「コンピューターを参照してドライバーを検索」→「参照」で用意した FX3\Win11\x64 フォルダを選択 →「次へ」
  5. 完了後、もう一度プロパティの「ドライバー」タブを開き、バージョンが 1.3.0.4 になっていることを確認して判定シートに記録
  6. デバイスが「！」マーク無しで表示されていることを確認（「！」が付いたら下の「困ったとき」へ）

ロールバック（新 → 旧）

  1. 同じ手順でプロパティの「ドライバー」タブを開く
  2. 「ドライバーを元に戻す」ボタンが押せる場合：クリックして案内に従う（理由は「以前のバージョンの方が安定していた」等でよい）
  3. ボタンが灰色で押せない場合：「ドライバーの更新」→「コンピューターを参照」→「コンピューター上の利用可能なドライバーの一覧から選択」→ 一覧から 1.2.3.20 系を選択 →「次へ」。一覧に無ければ「参照」で旧ドライバのフォルダを指定
  4. バージョンが 1.2.3.20 系に戻ったことを確認
  5. 復帰スモーク×1：AnalogBoard アプリを起動し、低密度1サイクルで Type C（ep6Timeouts=0・DDR_RD_END=1）を確認。結果を判定シート 3-4 に記録

困ったとき

  • デバイスに「！」が付いた／認識されない → USB コネクタを抜き差し。直らなければ PC を再起動
  • 更新ウィザードで「最適なドライバーが既にインストールされています」と出る → 「コンピューター上の利用可能なドライバーの一覧から選択」の経路でバージョンを明示指定する
  • どのデバイスか自信がない → デバイスを右クリック→プロパティ→「詳細」タブ→「ハードウェア ID」に VID_04B4 が含まれていれば対象

実験3：SDK1.3対応バイナリ検証（新ドライバ系列の最初に実施・レジストリ不要）

実験2より先にやる（2026-07-06 オーナー方針）

本実験はレジストリ操作が一切不要。これが成功すれば実験2（DFX A/B＝レジストリ変更）は丸ごとスキップできるため、新ドライバ系列では本実験を先に実施する。

目的判定基準（プラン確定）
新ドライバ＋一致SDK（1.3）のバイナリで通信可能かを検証し、ドライバ Plan A を確定させる Win11 新ドライバで Type C を達成

手順（driver_next.md の first gate を踏襲）

  1. 「ドライバ入替・ロールバック手順」に従い新ドライバ 1.3.0.4 へ入替（DFX 等の設定は一切変更しない＝既定のまま）
  2. 再作成した feature/win11-driver-compat の x64/Release 出力物を配置
  3. USBBoard_Connect の成功を確認（起動時自動接続のログ）
  4. first gate：アイドル EP4 読取を10回以上反復し、Get ep4 register data failed.／USB board is removed! Disconnect usb. が出ないことを確認
  5. first gate 通過後、EP2 → EP4 → EP6 の順で smoke：Set（EP2一括書込）→ レジスタ読取 → 低密度1サイクル取得
  6. 低密度で Type C を確認後、高密度×1〜3 で timeout と EP4 failure を再評価
  7. first gate が失敗した場合のみ実験2（DFX A/B）へ進む。成功した場合は実験2をスキップし判定シートに「スキップ（実験3成功）」と記録
  8. 系列の締めは「ドライバ入替・ロールバック手順」のロールバック＋復帰スモーク（実験2/3 が良好で新ドライバ維持を選ぶ場合はその判断を判定シート 3-4 に明記）

観測ポイント

  • endpoint discovery の変化有無（DLL に discovery hardening 導入済み）

実験2：Win11 DFX無効化 A/B（条件付き・スクリプトで実施）

実施条件（2026-07-06 オーナー方針：レジストリ操作は極力避ける）

実験3の first gate（接続＋アイドルEP4）が失敗した場合のみ実施する。実験3が成功していたら本実験はスキップし、判定シートに「スキップ（実験3成功）」と記録する。実施する場合もregedit は開かない——配布スクリプト（scripts\field-session\）が対象デバイスの自動検出・設定・復元まで行う。

目的判定基準（プラン確定）
Win11 新ドライバでの通信不可の主因が DFX（アイドルサスペンド）かを単変数で切り分け NoDfx（WdfDirectedPowerTransitionEnable=0）で Get ep4 register data failed. が消えるか
やってはいけないこと
  • INF（cyusb3.inf）を編集しない——署名 cat が無効化される
  • INF 同梱の CyUsb3NoDfx セクションでインストールしない——DriverEnableDfx=0 と WdfDirectedPowerTransitionEnable=0 の2変数が同時に変わり交絡する
  • regedit で手動編集しない——キー階層の打ち間違いリスク。必ず配布スクリプトを使う

事前準備（開発側・セッション前日まで）

  • ☐ scripts\field-session\ フォルダ一式（dfx_status.ps1／dfx_off.ps1／dfx_on.ps1／README）を実機PCにコピーしておく（例：C:\field-session\）

手順（クリック単位）

  1. 実施判断：実験3 first gate の失敗を判定シートで確認（失敗していなければここで終了・スキップ記録）
  2. 管理者ターミナルを開く：画面左下のスタートボタンを右クリック→「ターミナル (管理者)」（または「Windows PowerShell (管理者)」）→ 確認ダイアログで「はい」
  3. スクリプトのフォルダへ移動：cd C:\field-session（実際の配置先に読み替え）
  4. 現状確認：powershell -ExecutionPolicy Bypass -File .\dfx_status.ps1
    → 表示された値（通常「not set (driver default = DFX ENABLED)」）を判定シートに書き写す
  5. ベースライン（A条件・DFX有効のまま）：実験3と同じ first gate 試験（接続→アイドルEP4×10回）を実施し、失敗回数を記録——実験3の結果をそのまま流用してよい
  6. DFX 無効化（B条件）：powershell -ExecutionPolicy Bypass -File .\dfx_off.ps1
    → 緑色の「Done.」が出たら dfx_status.ps1 をもう一度実行し「0 = DFX DISABLED」を確認。黄色の「WARN: device restart failed」が出た場合はUSBコネクタを一度抜き差しすれば同じ効果
  7. B条件の試験：手順5と同一の first gate 試験を同一回数実施し、失敗回数を比較（判定 2-2）。失敗が消えたら、NoDfx のまま実験3の残り（smoke→低密度→高密度）を続行
  8. 復元（可逆性確認を兼ねる）：試験終了後 powershell -ExecutionPolicy Bypass -File .\dfx_on.ps1 を実行し、dfx_status.ps1 で「not set」に戻ったことを確認（判定 2-3）。※NoDfx を恒久運用にする判断をした場合は戻さず、その旨を判定シートに明記
  9. 全ステップの実行結果（スクリプト出力の要点）を判定シートのメモ欄に記録

スクリプトが失敗するとき

  • 「NG: administrator rights required」→ 手順2の管理者ターミナルで開き直す（ウィンドウタイトルに「管理者」と出ているか確認）
  • 「NG: FX3 board not found」→ USB 接続と機械の電源を確認して再実行
  • 赤字で「スクリプトの実行が無効」系のエラー → コマンドを powershell -ExecutionPolicy Bypass -File .\〜 の形で打っているか確認

実験4：r18高密度フィールドテスト（段階ラダー）

目的判定基準（プラン確定）

手順（段階制——各段でPassしてから次へ）
